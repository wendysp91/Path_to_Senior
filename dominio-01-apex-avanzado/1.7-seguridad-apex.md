## Por qué es crítico para un senior

El código Apex puede **bypassear todos los mecanismos de seguridad declarativos** de Salesforce si no se escribe con cuidado. Un senior entiende exactamente qué respeta y qué ignora cada keyword y método.

---

## `with sharing` / `without sharing` / `inherited sharing`

Estos keywords controlan **acceso a registros** — qué registros puede ver el código según las reglas de sharing del usuario. **No controlan FLS ni CRUD** — eso es separado.

### `with sharing` — aplica las reglas del usuario actual

```apex
public with sharing class ContactController {
    public List<Contact> getMyContacts() {
        // Solo devuelve contacts que el usuario actual puede ver
        // según OWD, role hierarchy y sharing rules
        return [SELECT Id, Name FROM Contact];
    }
}
```

### `without sharing` — ignora todas las reglas de sharing

```apex
public without sharing class ContactAdminService {
    public List<Contact> getAllContacts() {
        // Devuelve TODOS los contacts — útil para jobs de sistema
        return [SELECT Id, Name FROM Contact];
    }
}
```

### `inherited sharing` — hereda el contexto del caller

```apex
public inherited sharing class ContactService {
    public List<Contact> getContacts() {
        // Si me llama un with sharing → aplica sharing
        // Si me llama un without sharing → ignora sharing
        return [SELECT Id, Name FROM Contact];
    }
}
```

### Sin keyword — el más peligroso

```apex
public class ContactService {
    // SIN keyword → actúa como without sharing en la mayoría de contextos
    // Es una vulnerabilidad silenciosa — NUNCA hagas esto
}
```

### Cuándo usar cada uno

```
Controllers / LWC Apex methods    →  with sharing
Service Layer                     →  inherited sharing (flexible)
Batch Apex / Scheduled Jobs       →  without sharing
Admin utilities / Data migration  →  without sharing
REST API inbound handlers         →  with sharing o inherited sharing
```

**Regla de arquitectura:** La capa de entrada define el contexto de sharing. Las capas internas usan `inherited sharing` para ser reutilizables en ambos contextos.

## CRUD y FLS — dos conceptos distintos

| Concepto | Qué controla |
|---|---|
| **Sharing** | Qué **registros** puede ver el usuario |
| **CRUD** | Si puede crear/leer/actualizar/borrar ese **tipo de objeto** |
| **FLS** | Si puede ver/editar un **campo específico** |

`with sharing` solo controla sharing. Un usuario sin permiso de Create en Contact puede igualmente insertar un Contact si tu código no chequea CRUD.

---

## La sintaxis moderna — `WITH USER_MODE` y `as user`

Desde Spring 2023. Enforces CRUD + FLS + sharing de forma automática:

```apex
// SOQL que respeta sharing, CRUD y FLS automáticamente
List<Contact> contacts = [SELECT Id, Email FROM Contact WITH USER_MODE];

// SOQL explícitamente en modo sistema
List<Contact> all = [SELECT Id, Email FROM Contact WITH SYSTEM_MODE];

// DML que respeta CRUD y FLS automáticamente
insert as user newContacts;
update as user modifiedContacts;
delete as user oldContacts;
upsert as user records;

// DML explícitamente en modo sistema
insert as system records;
```

**Este es el estándar para código nuevo.**

## `Security.stripInaccessible()` — el estándar anterior, aún muy presente

```apex
// Para lectura — elimina campos que el usuario no puede leer
SObjectAccessDecision decision = Security.stripInaccessible(
    AccessType.READABLE,
    [SELECT Id, Email, Salary__c FROM Contact]
);
List<Contact> safeContacts = (List<Contact>) decision.getRecords();

// Para inserción — elimina campos que el usuario no puede crear
SObjectAccessDecision createDecision = Security.stripInaccessible(
    AccessType.CREATABLE,
    newContacts
);
insert createDecision.getRecords();

// Para actualización
SObjectAccessDecision updateDecision = Security.stripInaccessible(
    AccessType.UPDATABLE,
    contactsToUpdate
);
update updateDecision.getRecords();
```

### AccessType options

| AccessType | Aplica a |
|---|---|
| `READABLE` | Queries, datos enviados al usuario |
| `CREATABLE` | INSERT |
| `UPDATABLE` | UPDATE |
| `UPSERTABLE` | UPSERT |

## Resumen — qué enforza qué

```
                    Sharing     CRUD      FLS
─────────────────────────────────────────────────
with sharing           ✓          ✗        ✗
without sharing        ✗          ✗        ✗
WITH USER_MODE         ✓          ✓        ✓
WITH SYSTEM_MODE       ✗          ✗        ✗
as user (DML)          —          ✓        ✓
as system (DML)        —          ✗        ✗
stripInaccessible()    ✗          ✗        ✓
```

### La combinación correcta para código orientado al usuario

```apex
public with sharing class MyController {
    public List<Contact> getContacts() {
        return [SELECT Id, Name, Email FROM Contact WITH USER_MODE];
    }

    public void saveContact(Contact c) {
        insert as user c;
    }
}
```

## Apex Managed Sharing

Cuando las sharing rules declarativas no son suficientes. Cada objeto shareable tiene un objeto Share:

```
Account     →  AccountShare
Contact     →  ContactShare
MiObj__c    →  MiObj__Share
```

### Crear un share

```apex
// Dar acceso de lectura a un usuario sobre un Contact específico
ContactShare share = new ContactShare(
    ContactId          = contactId,
    UserOrGroupId      = userId,
    ContactAccessLevel = 'Read',     // 'Read' o 'Edit'
    RowCause           = Schema.ContactShare.RowCause.Manual
);
insert share;

// Compartir con un Public Group (más escalable)
ContactShare groupShare = new ContactShare(
    ContactId          = contactId,
    UserOrGroupId      = groupId,
    ContactAccessLevel = 'Edit',
    RowCause           = Schema.ContactShare.RowCause.Manual
);
insert groupShare;
```

### RowCause — el concepto más importante

`RowCause` es la razón por la que existe el share. Puedes crear **causas personalizadas** en el objeto para gestionar tus shares de forma independiente.

```apex
// Custom Share Reason "Regional_Access__c" definido en el objeto

// Primero elimina los shares de TU causa (sin afectar los demás)
delete [
    SELECT Id FROM Contact__Share
    WHERE ParentId = :recordId
    AND RowCause = :Schema.Contact__Share.RowCause.Regional_Access__c
];

// Luego inserta los nuevos
insert new Contact__Share(
    ParentId      = recordId,
    UserOrGroupId = regionGroupId,
    AccessLevel   = 'Read',
    RowCause      = Schema.Contact__Share.RowCause.Regional_Access__c
);
```

### Apex Managed Sharing requiere `without sharing`

```apex
public without sharing class SharingService {
    public static void shareWithRegion(Id recordId, Id regionGroupId) {
        Contact__Share share = new Contact__Share(
            ParentId      = recordId,
            UserOrGroupId = regionGroupId,
            AccessLevel   = 'Read',
            RowCause      = Schema.Contact__Share.RowCause.Regional_Access__c
        );

        try {
            insert share;
        } catch (DmlException e) {
            if (!e.getMessage().contains('DUPLICATE_VALUE')) {
                throw e;
            }
        }
    }
}
```

## Seguridad en el contexto de integración

### El endpoint REST debe respetar el usuario de la integración

```apex
@RestResource(urlMapping='/api/contacts/*')
global with sharing class ContactRESTHandler {

    @HttpPost
    global static ResponseWrapper createContact(ContactPayload payload) {
        Contact c = new Contact(
            LastName = payload.lastName,
            Email    = payload.email
        );

        insert as user c; // enforza CRUD y FLS del integration user

        return new ResponseWrapper(c.Id, 'created');
    }
}
```

### Nunca confíes en el input externo

```apex
// El sistema externo envía un field name dinámico
String fieldName = payload.get('fieldName');

// MAL — usas el fieldName directo
String query = 'SELECT ' + fieldName + ' FROM Contact';

// BIEN — validas contra el schema antes de usar
Map<String, Schema.SObjectField> validFields =
    Schema.SObjectType.Contact.fields.getMap();

if (!validFields.containsKey(fieldName.toLowerCase())) {
    throw new ValidationException('Campo no válido: ' + fieldName);
}
```

## Ejercicio práctico

Diseña e implementa un sistema de acceso regional:

**Contexto:** Objeto `Client__c` con OWD = Private. Los usuarios solo ven clients de su región. Los gerentes regionales ven todos los clients de su región aunque no sean dueños.

**Tu solución debe:**
1. `ClientController` (with sharing) que devuelve clients al usuario usando `WITH USER_MODE`
2. `RegionalSharingService` (without sharing) que:
   - Al insertar/actualizar un `Client__c`, comparte con el Public Group de su región
   - Usa un **Custom Share Reason** `Regional_Access__c`
   - Al cambiar la región, elimina los shares de la región anterior y crea los nuevos
3. Trigger en `Client__c` que llama al Service en `after insert` y `after update`
4. Tests que verifican:
   - Un usuario de Región Norte ve clients de Norte pero no de Sur
   - Al cambiar la región, el usuario de Sur ya no lo ve y el de Norte sí
