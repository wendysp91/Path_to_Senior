# 1.2 — Governor Limits

[← Volver al índice](../README.md) | [← 1.1 Asincronía](1.1-asincronia.md)

---

## Por qué existen

Salesforce es multi-tenant — miles de orgs comparten el mismo servidor. Los Governor Limits impiden que una org acapare recursos y afecte a las demás. No son opcionales. Un senior los tiene internalizados — no los memoriza, los **piensa** mientras diseña.

## Cómo se resetean los límites

Cada **contexto de ejecución** tiene su propio contador desde cero.

```
Usuario guarda un registro
│
├─ Trigger before insert     → contexto 1 (100 SOQL disponibles)
├─ Trigger after insert      → MISMO contexto 1 (comparte los 100)
│
│  El trigger llama @future → contexto 2 (100 SOQL frescos)
│
│  El callout dispara webhook → contexto 3 (100 SOQL frescos)
```

**La regla:** Async = nueva transacción = límites frescos. Dentro de una misma transacción, todos los triggers, clases y métodos comparten el mismo contador.

**Caso trampa:** Un trigger before + after + un trigger en un objeto relacionado que se actualiza en el mismo save — todo comparte límites.

## La excepción que NO puedes atrapar

```apex
// LimitException NO se puede atrapar — Salesforce mata la transacción antes
try {
    // 101 queries en transacción síncrona...
} catch (LimitException e) {
    // NUNCA llega aquí
}
```

Los límites de gobernador son la única excepción real que no puedes manejar. Por eso el diseño bulk-safe no es opcional.

## Los dos límites que más golpean en integración

### CPU Time (10 segundos en sync)

No mide tiempo de reloj — mide solo tiempo de procesador. Los callouts y waits no cuentan.

**Lo que más consume CPU:**
- Loops grandes con lógica compleja
- JSON parsing de payloads grandes
- String concatenation en loops

```apex
// MATA el CPU en listas grandes
String result = '';
for (String item : hugeList) {
    result += item + ','; // crea un nuevo String object en cada iteración
}

// CORRECTO
List<String> parts = new List<String>();
for (String item : hugeList) {
    parts.add(item);
}
String result = String.join(parts, ',');
```

### Heap Size (6 MB sync / 12 MB async)

**Estrategia:** Solo haz `SELECT` de los campos que necesitas. Nunca selecciones todos los campos cuando solo necesitas dos.

## El error más común: SOQL dentro de un loop

```apex
// MATA tus límites — un SOQL por cada registro
for (Account acc : accounts) {
    List<Contact> contacts = [SELECT Id FROM Contact WHERE AccountId = :acc.Id];
}

// CORRECTO — un solo SOQL para todos
Set<Id> accountIds = new Map<Id, Account>(accounts).keySet();
Map<Id, List<Contact>> contactsByAccount = new Map<Id, List<Contact>>();

for (Contact c : [SELECT Id, AccountId FROM Contact WHERE AccountId IN :accountIds]) {
    if (!contactsByAccount.containsKey(c.AccountId)) {
        contactsByAccount.put(c.AccountId, new List<Contact>());
    }
    contactsByAccount.get(c.AccountId).add(c);
}

for (Account acc : accounts) {
    List<Contact> contacts = contactsByAccount.get(acc.Id); // cero SOQL
}
```

Lo mismo aplica para DML:

```apex
// MAL — DML en loop
for (Contact c : contacts) {
    c.Status__c = 'Processed';
    update c;
}

// BIEN — un solo DML
for (Contact c : contacts) {
    c.Status__c = 'Processed';
}
update contacts;
```

## Regla de oro de integración: no puedes hacer callouts después de DML

```
┌─────────────────────────────────────────────────────────────┐
│ No puedes hacer callouts si ya hiciste DML en la misma     │
│ transacción (uncommitted work pending)                      │
└─────────────────────────────────────────────────────────────┘
```

```apex
// FALLA
insert miRegistro;
h.send(request); // ← ERROR: uncommitted work pending

// SOLUCIÓN 1: Queueable (nueva transacción)
insert miRegistro;
System.enqueueJob(new CalloutJob(miRegistro.Id));

// SOLUCIÓN 2: Callout primero, DML después
HttpResponse res = h.send(request);
// procesar respuesta...
insert miRegistro;
```

## Estrategias de diseño

### 1. Piensa en el peor caso antes de escribir

¿Cuántos registros puede recibir esto en el peor caso? Si la respuesta es "podría ser miles", usa Batch desde el inicio.

### 2. `Map<Id, SObject>` sobre listas para lookups

```apex
Map<Id, Account> accountMap = new Map<Id, Account>([SELECT Id, Name FROM Account WHERE Id IN :ids]);
Account acc = accountMap.get(someId); // O(1) — no itera la lista
```

### 3. SOQL for loops para volumen grande

```apex
// Carga todo en memoria — puede reventar heap
List<Contact> todos = [SELECT Id FROM Contact]; // peligroso con 200k registros

// SOQL for loop — Salesforce pagina internamente, menor heap
for (Contact c : [SELECT Id FROM Contact]) {
    // procesa internamente de a 200 registros
}
```

### 4. Agrupa callouts — diseña endpoints bulk

```
// MAL: 200 callouts para 200 registros
POST /api/contact/1
POST /api/contact/2
...

// BIEN: 1 callout para 200 registros
POST /api/contacts
Body: [{id:1,...}, {id:2,...}, ...]
```

### 5. Usa Platform Cache para evitar queries repetidos

```apex
// En lugar de hacer el mismo SOQL N veces por transacción
Cache.Org.put('local.myPartition.config', configData, 3600);
Object cached = Cache.Org.get('local.myPartition.config');
```

## Ejercicio práctico

Dado este código con 5 errores de límites, encuéntralos y corrígelos:

```apex
public class OrderProcessor {
    public static void processOrders(List<Order__c> orders) {
        for (Order__c order : orders) {
            // Error 1
            Account acc = [SELECT Id, Name FROM Account WHERE Id = :order.Account__c];

            List<Product__c> products = [SELECT Id, Price__c FROM Product__c
                                         WHERE Order__c = :order.Id];
            String summary = '';
            for (Product__c p : products) {
                // Error 2
                summary += p.Id + ',' + p.Price__c + '|';
            }

            order.Summary__c = summary;
            order.Account_Name__c = acc.Name;

            // Error 3
            update order;
        }

        for (Order__c order : orders) {
            Http h = new Http();
            HttpRequest req = new HttpRequest();
            req.setEndpoint('callout:ExternalSystem/orders');
            req.setMethod('POST');
            req.setBody(JSON.serialize(order));
            // Error 4 (loop) + Error 5 (DML antes de callout)
            h.send(req);
        }
    }
}
```
