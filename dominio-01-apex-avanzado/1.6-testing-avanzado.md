## Por qué el testing define tu nivel

Un junior escribe tests para llegar al 75% de cobertura. Un senior escribe tests que **detectan bugs antes de que lleguen a producción**. La diferencia no es cuántas líneas cubres — es qué comportamientos verificas. Los tests son documentación ejecutable de cómo debe funcionar el sistema.

---

## `Test.startTest()` / `Test.stopTest()` — qué hace realmente

### Función 1: Resetea governor limits

```apex
@isTest
static void miTest() {
    // Aquí gastas limits del contexto de setup
    List<Contact> contacts = [SELECT Id FROM Contact]; // gasta queries del test

    Test.startTest();
    // Aquí tienes limits FRESCOS — 100 SOQL nuevos, 150 DML nuevos
    ContactService.processAll(contacts);
    Test.stopTest();

    // Aquí verificas resultados
}
```

### Función 2: Fuerza ejecución síncrona del código async

```apex
@isTest
static void testBatchJob() {
    // crear datos...

    Test.startTest();
    Database.executeBatch(new ContactMigrationBatch(), 200);
    Test.stopTest();
    // stopTest() ESPERA a que el batch complete

    // Ahora puedes verificar el resultado del batch
    List<Contact> processed = [SELECT Id, Migrated__c FROM Contact];
    for (Contact c : processed) {
        System.assertEquals(true, c.Migrated__c, 'Contacto debe estar marcado como migrado');
    }
}
```

### Lo que NO hace en tests

- Los jobs Queueable encadenados (el hijo del hijo) **no corren** — solo el primero
- Si tu lógica depende de una cadena de Queueables, testea cada eslabón por separado

## Test Data Factories

### Patrón simple — método por objeto

```apex
@isTest
public class TestDataFactory {

    public static Account createAccount(Boolean doInsert) {
        Account acc = new Account(
            Name           = 'Test Account',
            BillingCountry = 'Mexico',
            Industry       = 'Technology'
        );
        if (doInsert) insert acc;
        return acc;
    }

    public static Contact createContact(Id accountId, Boolean doInsert) {
        Contact c = new Contact(
            LastName  = 'Test',
            Email     = 'test@example.com',
            AccountId = accountId
        );
        if (doInsert) insert c;
        return c;
    }

    // Para volumen — bulkificación en tests
    public static List<Contact> createContacts(Integer quantity, Id accountId, Boolean doInsert) {
        List<Contact> contacts = new List<Contact>();
        for (Integer i = 0; i < quantity; i++) {
            contacts.add(new Contact(
                LastName  = 'Test ' + i,
                Email     = 'test' + i + '@example.com',
                AccountId = accountId
            ));
        }
        if (doInsert) insert contacts;
        return contacts;
    }
}
```

### Patrón builder — más flexible y expresivo

```apex
@isTest
public class ContactBuilder {
    private Contact record = new Contact(
        LastName = 'Default',
        Email    = 'default@test.com'
    );

    public ContactBuilder withLastName(String lastName) {
        record.LastName = lastName;
        return this;
    }
    public ContactBuilder withEmail(String email) {
        record.Email = email;
        return this;
    }
    public ContactBuilder withAccount(Id accountId) {
        record.AccountId = accountId;
        return this;
    }
    public ContactBuilder asInactive() {
        record.Active__c = false;
        return this;
    }

    public Contact build()          { return record; }
    public Contact buildAndInsert() { insert record; return record; }
}

// En el test — legible, sin ambigüedad
Contact contact = new ContactBuilder()
    .withLastName('García')
    .withEmail('garcia@example.com')
    .asInactive()
    .buildAndInsert();
```

## `@testSetup` — cuándo usarlo y sus limitaciones

```apex
@isTest
public class ContactServiceTest {

    @testSetup
    static void setupData() {
        // Corre UNA VEZ para toda la clase
        Account acc = TestDataFactory.createAccount(true);
        TestDataFactory.createContacts(200, acc.Id, true);
    }

    @isTest
    static void testProcessActive() {
        // Cada test EMPIEZA con los 200 contacts del setup — limpios
        List<Contact> contacts = [SELECT Id FROM Contact WHERE Active__c = true];

        Test.startTest();
        ContactService.processActive(contacts);
        Test.stopTest();
    }

    @isTest
    static void testProcessInactive() {
        // Los cambios de testProcessActive NO están aquí — cada test es independiente
        List<Contact> contacts = [SELECT Id FROM Contact WHERE Active__c = false];
    }
}
```

### Limitaciones de `@testSetup`

```
❌ No puedes retornar valores — debes re-querear en cada test
❌ No funciona con @isTest(SeeAllData=true)
❌ Si @testSetup falla, TODOS los tests de la clase fallan
❌ DML en @testSetup cuenta contra los limits del setup, no del test
✓ Úsalo cuando los datos son iguales para todos los tests de la clase
✗ No lo uses si cada test necesita datos muy diferentes
```

## Mocking de HTTP Callouts

### Método 1: `HttpCalloutMock` — implementación propia (más control)

```apex
public class ContactAPISuccessMock implements HttpCalloutMock {
    public HTTPResponse respond(HTTPRequest request) {
        // Puedes verificar el request aquí
        System.assertEquals('POST', request.getMethod());
        System.assert(request.getEndpoint().contains('/contacts'));

        HttpResponse res = new HttpResponse();
        res.setStatusCode(200);
        res.setHeader('Content-Type', 'application/json');
        res.setBody('{"id": "ext-123", "status": "created"}');
        return res;
    }
}

public class ContactAPIErrorMock implements HttpCalloutMock {
    public HTTPResponse respond(HTTPRequest request) {
        HttpResponse res = new HttpResponse();
        res.setStatusCode(500);
        res.setBody('{"error": "Internal Server Error"}');
        return res;
    }
}

// En el test
@isTest
static void testCalloutSuccess() {
    Test.setMock(HttpCalloutMock.class, new ContactAPISuccessMock());

    Test.startTest();
    ContactIntegrationService.syncContact('003xxx');
    Test.stopTest();

    // verificar que el registro se actualizó correctamente
}

@isTest
static void testCalloutError() {
    Test.setMock(HttpCalloutMock.class, new ContactAPIErrorMock());

    Test.startTest();
    ContactIntegrationService.syncContact('003xxx');
    Test.stopTest();

    List<Integration_Error__c> errors = [SELECT Id FROM Integration_Error__c];
    System.assertEquals(1, errors.size(), 'Debe haber un error loggeado');
}
```

### Método 2: `StaticResourceCalloutMock` — respuesta desde Static Resource

Ideal cuando el JSON de respuesta es grande:

```apex
StaticResourceCalloutMock mock = new StaticResourceCalloutMock();
mock.setStaticResource('MockContactAPIResponse'); // nombre del Static Resource
mock.setStatusCode(200);
mock.setHeader('Content-Type', 'application/json');
Test.setMock(HttpCalloutMock.class, mock);
```

### Método 3: `MultiStaticResourceCalloutMock` — respuestas diferentes por endpoint

```apex
MultiStaticResourceCalloutMock multiMock = new MultiStaticResourceCalloutMock();

multiMock.setStaticResource('https://api.erp.com/contacts', 'MockERPContactResponse');
multiMock.setStaticResource('https://api.erp.com/orders',   'MockERPOrderResponse');

Test.setMock(HttpCalloutMock.class, multiMock);
```

## `System.StubProvider` — mocking nativo de clases

```apex
// La dependencia a mockear
public interface ExternalCRMService {
    Map<String, Object> getContactData(String externalId);
    Boolean updateContact(String externalId, Map<String, Object> data);
}

// El stub — intercepta las llamadas al método
@isTest
public class ExternalCRMServiceStub implements System.StubProvider {

    public Map<String, Object> capturedData;
    public Boolean shouldReturnError = false;

    public Object handleMethodCall(
        Object       stubbedObject,
        String       stubbedMethodName,
        Type         returnType,
        List<Type>   listOfParamTypes,
        List<String> listOfParamNames,
        List<Object> listOfArgs
    ) {
        if (stubbedMethodName == 'getContactData') {
            if (shouldReturnError) return null;
            return new Map<String, Object>{'name' => 'García', 'email' => 'garcia@test.com'};
        }
        if (stubbedMethodName == 'updateContact') {
            capturedData = (Map<String, Object>) listOfArgs[1];
            return true;
        }
        return null;
    }
}

// En el test
@isTest
static void testSyncUsesCorrectData() {
    ExternalCRMServiceStub stub = new ExternalCRMServiceStub();
    ExternalCRMService mockCRM = (ExternalCRMService) Test.createStub(
        ExternalCRMService.class,
        stub
    );

    ContactSyncService service = new ContactSyncService(mockCRM);

    Test.startTest();
    service.sync('ext-123');
    Test.stopTest();

    System.assertNotNull(stub.capturedData, 'updateContact debe haber sido llamado');
    System.assertEquals('García', stub.capturedData.get('name'));
}
```

**Limitación:** Solo funciona con interfaces o clases no-final con métodos no-estáticos.

## `Test.isRunningTest()` — cuándo sí y cuándo no

### El anti-patrón

```apex
// MAL — el código en test ≠ código en producción
public void syncToERP(List<Contact> contacts) {
    if (Test.isRunningTest()) {
        return; // skip en tests — no estás probando nada útil
    }
    // callout real...
}
```

Si usas esto para saltarte callouts, el test no prueba nada. La solución correcta es `Test.setMock()`.

### Cuándo sí es aceptable

```apex
// OK — datos que deben ser determinísticos en tests
public String generateSessionToken() {
    if (Test.isRunningTest()) {
        return 'TEST-TOKEN-FIXED';
    }
    return EncodingUtil.convertToHex(Crypto.generateAESKey(128));
}

// OK — evitar jobs schedulados que contaminan la org de test
public static void scheduleNightlyJob() {
    if (!Test.isRunningTest()) {
        System.schedule('Nightly Sync', '0 0 2 * * ?', new NightlySyncJob());
    }
}
```

## Estrategia para 85%+ con tests que valen

### El patrón Arrange-Act-Assert (AAA)

```apex
@isTest
static void testEmailChangeResetsVerification() {
    // ARRANGE — preparar el estado inicial
    Contact contact = new ContactBuilder()
        .withEmail('original@test.com')
        .buildAndInsert();
    contact.Email_Verified__c = true;
    update contact;

    // ACT — ejecutar la acción a probar
    Test.startTest();
    contact.Email = 'nuevo@test.com';
    update contact;
    Test.stopTest();

    // ASSERT — verificar el resultado esperado
    Contact updated = [SELECT Email_Verified__c FROM Contact WHERE Id = :contact.Id];
    System.assertEquals(
        false,
        updated.Email_Verified__c,
        'Cambiar email debe resetear la verificación'
    );
}
```

### Escenarios que debes cubrir siempre

```
Para cada pieza de lógica:
├── Happy path — funciona con datos válidos normales
├── Bulk — funciona con 200 registros (el límite de un trigger)
├── Edge cases:
│   ├── Lista vacía
│   ├── Un solo registro
│   └── Valores null en campos opcionales
└── Casos de error:
    ├── Datos inválidos
    ├── Callout fallido (4xx, 5xx, timeout)
    └── DML fallido (constraints, duplicates)
```

### Cobertura que no vale nada

```apex
// Esto da cobertura pero no prueba NADA
@isTest
static void coverageTest() {
    ContactService.processAll(new List<Contact>());
    // sin assertions — solo ejecuta el código
}
```

Un test sin `System.assert` / `System.assertEquals` / `System.assertNotEquals` no es un test — es cobertura de mentira.

## Ejercicio práctico

Escribe el suite de tests completo para este servicio:

```apex
public class LeadEnrichmentService {
    private ExternalDataAPI api;

    public LeadEnrichmentService(ExternalDataAPI api) {
        this.api = api;
    }

    public void enrichLeads(List<Lead> leads) {
        List<Lead> toUpdate = new List<Lead>();

        for (Lead lead : leads) {
            if (String.isBlank(lead.Email)) continue;

            Map<String, Object> data = api.lookupByEmail(lead.Email);
            if (data == null) continue;

            lead.Company     = (String) data.get('company');
            lead.Industry    = (String) data.get('industry');
            lead.Enriched__c = true;
            toUpdate.add(lead);
        }

        if (!toUpdate.isEmpty()) {
            List<Database.SaveResult> results = Database.update(toUpdate, false);
            // manejar errores parciales...
        }
    }
}
```

**Tu suite debe cubrir:**
1. Lead enriquecido correctamente cuando la API devuelve datos
2. Lead sin email es ignorado (no se llama a la API)
3. API devuelve null — Lead no se actualiza
4. API falla — excepción controlada, otros leads siguen procesando
5. DML falla en algún lead — modo parcial, error loggeado, otros actualizados
6. Bulk — 200 leads procesados correctamente
7. Verificar que cuando `Enriched__c = true` el campo Company también cambió
