## Por qué importa para arquitectura

El manejo de excepciones es donde se ve la madurez de un sistema. Código que falla silenciosamente es peor que código que no funciona — no sabes cuándo ni por qué fallaste, y los datos quedan en estado inconsistente. Un senior diseña la **estrategia de errores antes de escribir la lógica feliz**.

---

## Tipos de excepciones en Apex

### Excepciones del sistema (built-in)

| Excepción | Cuándo ocurre |
|---|---|
| `DmlException` | Error en INSERT/UPDATE/DELETE (constraint, required field) |
| `QueryException` | SOQL devuelve 0 rows con `getSingleResult()`, o más de 1 con assignment directo |
| `CalloutException` | Timeout, error de red, respuesta inválida en HTTP |
| `NullPointerException` | Acceder a variable o propiedad de un null |
| `TypeException` | Cast inválido |
| `JSONException` | JSON malformado o tipo inesperado |
| `SObjectException` | Acceder a campo no incluido en el SELECT |
| `ListException` | Índice fuera de rango en una lista |
| `MathException` | División entre cero |
| `AsyncException` | Error en operación async (encolar job al límite) |
| `SecurityException` | Violación de permisos |

### La excepción que NO puedes atrapar

```apex
// LimitException NO se puede atrapar — Salesforce mata la transacción antes
try {
    // 101 queries en transacción síncrona
} catch (LimitException e) {
    // NUNCA llega aquí
}
```

**Implicación:** Los límites de gobernador son la única excepción real que no puedes manejar. El diseño bulk-safe es la única defensa.

## Custom Exceptions — diseña una jerarquía

```apex
// Excepción base de tu aplicación/integración
public class AppException extends Exception {}

// Categorías de error
public class IntegrationException    extends AppException {}
public class ValidationException     extends AppException {}
public class ConfigurationException  extends AppException {}
public class AuthenticationException extends IntegrationException {}

// Uso
throw new IntegrationException('El sistema externo devolvió 503');
throw new ValidationException('Email inválido: ' + email);
throw new AuthenticationException('Token expirado');
```

### Por qué una jerarquía y no solo `Exception`

```apex
try {
    callExternalSystem();
} catch (AuthenticationException e) {
    // token expirado — refrescar y reintentar
    refreshToken();
    callExternalSystem();
} catch (IntegrationException e) {
    // fallo de integración — loggear y continuar
    logError(e);
} catch (Exception e) {
    // cualquier otra cosa — relanzar
    throw e;
}
```

Puedes responder diferente a cada tipo de fallo.

## El anti-patrón más común — tragarse la excepción

```apex
// MAL — el sistema falla silenciosamente
try {
    Database.insert(records);
} catch (Exception e) {
    // vacío
}

// MAL — debug solo visible en logs, no en producción de forma útil
try {
    Database.insert(records);
} catch (Exception e) {
    System.debug('Error: ' + e.getMessage()); // nadie lo ve en producción
}

// CORRECTO
try {
    Database.insert(records);
} catch (DmlException e) {
    ErrorLogger.log('ContactService.insertRecords', e);
    throw e; // relanza si el caller necesita saber
}
```

## Excepciones en contextos asíncronos

### En `@future`

```apex
@future
public static void syncToERP(Set<Id> ids) {
    try {
        // lógica
    } catch (Exception e) {
        // si no capturas aquí, el job falla silenciosamente
        // no puedes notificar al caller — ya es otra transacción
        ErrorLogger.log('ERPSync.syncToERP', e);
    }
}
```

### En `Batch Apex`

```apex
public void execute(Database.BatchableContext bc, List<Contact> scope) {
    try {
        procesarLote(scope);
    } catch (Exception e) {
        // Falla solo este CHUNK — los demás chunks siguen
        ErrorLogger.log('ContactBatch.execute', e, bc.getJobId());
    }
}

public void finish(Database.BatchableContext bc) {
    AsyncApexJob job = [
        SELECT NumberOfErrors, JobItemsProcessed, TotalJobItems, Status
        FROM AsyncApexJob
        WHERE Id = :bc.getJobId()
    ];

    if (job.NumberOfErrors > 0) {
        Messaging.sendEmail(new List<Messaging.SingleEmailMessage>{
            buildErrorEmail(job)
        });
    }
}
```

### En `Queueable` con Finalizer — el patrón más robusto

```apex
public class SyncJob implements Queueable, Database.AllowsCallouts {
    public void execute(QueueableContext ctx) {
        System.attachFinalizer(new SyncFinalizer()); // registrar antes de cualquier código

        performSync(); // si esto lanza excepción, el Finalizer la captura
    }
}

public class SyncFinalizer implements Finalizer {
    public void execute(FinalizerContext ctx) {
        if (ctx.getResult() == ParentJobResult.UNHANDLED_EXCEPTION) {
            Exception e = ctx.getException();

            ErrorLogger.log(
                'SyncJob',
                e.getMessage(),
                e.getStackTraceString(),
                ctx.getAsyncApexJobId()
            );

            if (debeReintentar(ctx)) {
                System.enqueueJob(new SyncJob());
            }
        }
    }
}
```

## DML Parcial — `Database.SaveResult`

```apex
List<Database.SaveResult> results = Database.insert(contacts, false); // false = allOrNone = false

List<Integration_Error__c> errorLogs = new List<Integration_Error__c>();

for (Integer i = 0; i < results.size(); i++) {
    if (!results[i].isSuccess()) {
        for (Database.Error err : results[i].getErrors()) {
            errorLogs.add(new Integration_Error__c(
                Related_Record_Id__c  = contacts[i].Id,
                Error_Status_Code__c  = String.valueOf(err.getStatusCode()),
                Error_Message__c      = err.getMessage(),
                Failed_Fields__c      = String.join(err.getFields(), ', '),
                Context__c            = 'ContactService.insertContacts'
            ));
        }
    }
}

if (!errorLogs.isEmpty()) insert errorLogs;
```

### Equivalentes para otros tipos de DML

```apex
// Upsert
List<Database.UpsertResult> upsertResults = Database.upsert(records, ExternalId__c, false);
Boolean wasCreated = upsertResults[0].isCreated(); // true=insert, false=update

// Delete
List<Database.DeleteResult> deleteResults = Database.delete(records, false);

// Update
List<Database.SaveResult> updateResults = Database.update(records, false);
```

## El sistema de logging

### Patrón 1: Objeto de log directo (simple, síncrono)

```apex
public class ErrorLogger {

    public static void log(String context, Exception e) {
        log(context, e.getMessage(), e.getStackTraceString(), e.getTypeName());
    }

    public static void log(String context, String message, String stackTrace, String errorType) {
        try {
            insert new Integration_Error__c(
                Context__c        = context,
                Error_Message__c  = message,
                Stack_Trace__c    = stackTrace,
                Error_Type__c     = errorType,
                Timestamp__c      = Datetime.now(),
                User__c           = UserInfo.getUserId(),
                Transaction_Id__c = Request.getCurrent().getRequestId()
            );
        } catch (Exception logError) {
            System.debug(LoggingLevel.ERROR,
                'LOGGING FAILED: ' + logError.getMessage() + ' | Original: ' + message);
        }
    }
}
```

**Problema:** Si la transacción principal hace rollback, el log también se pierde.

### Patrón 2: Platform Events para logging (robusto)

El log sobrevive al rollback de la transacción principal.

```apex
public class ErrorLogger {

    public static void log(String context, Exception e) {
        Error_Log__e errorEvent = new Error_Log__e(
            Context__c        = context,
            Error_Message__c  = e.getMessage(),
            Stack_Trace__c    = e.getStackTraceString(),
            Error_Type__c     = e.getTypeName(),
            User_Id__c        = UserInfo.getUserId()
        );

        // publish() sobrevive al rollback de la transacción principal
        EventBus.publish(errorEvent);
    }
}

// Un trigger sobre el Platform Event guarda el log
trigger ErrorLogTrigger on Error_Log__e (after insert) {
    List<Integration_Error__c> logs = new List<Integration_Error__c>();
    for (Error_Log__e event : Trigger.new) {
        logs.add(new Integration_Error__c(
            Context__c        = event.Context__c,
            Error_Message__c  = event.Error_Message__c,
            Stack_Trace__c    = event.Stack_Trace__c,
            Error_Type__c     = event.Error_Type__c,
            User__c           = event.User_Id__c
        ));
    }
    insert logs;
}
```

## Estrategia de excepciones por capa

```
┌─────────────────────────────────────────────────────────────────┐
│  TRIGGER / CONTROLLER (capa de entrada)                         │
│  → Captura todo, loggea, muestra mensaje amigable al usuario    │
│  → Nunca deja que una excepción llegue al usuario como stack    │
└─────────────────────────────┬───────────────────────────────────┘
                              │ relanza excepciones tipadas
┌─────────────────────────────▼───────────────────────────────────┐
│  SERVICE LAYER (lógica de negocio)                              │
│  → Captura excepciones de infraestructura (DML, callout)        │
│  → Las convierte en excepciones de dominio (IntegrationException)│
│  → Loggea con contexto de negocio                               │
└─────────────────────────────┬───────────────────────────────────┘
                              │ relanza excepciones de dominio
┌─────────────────────────────▼───────────────────────────────────┐
│  SELECTOR / INTEGRATION LAYER (datos e integración)             │
│  → Captura excepciones técnicas (CalloutException, DmlException) │
│  → Agrega contexto técnico al mensaje                           │
│  → Relanza como excepciones tipadas de tu jerarquía             │
└─────────────────────────────────────────────────────────────────┘
```

**La regla:** Las capas bajas convierten excepciones técnicas en excepciones de dominio. Las capas altas deciden qué hacer con ellas.

## Ejercicio práctico

Diseña un sistema completo de manejo de errores para un Batch que procesa 10,000 Contacts:

1. Si el callout falla por timeout → loggear y continuar con el siguiente (no cancelar chunk)
2. Si el callout falla por 401 Unauthorized → lanzar `AuthenticationException` y cancelar el chunk
3. Si el DML falla en algún registro → modo parcial, loggear los fallidos, continuar
4. El log debe sobrevivir aunque el chunk haga rollback → usa Platform Events
5. El `finish()` reporta: total procesados, exitosos, fallidos; notifica por email si hay más de 100 errores
6. Un `Finalizer` en un Queueable intermedio maneja el fallo catastrófico del batch completo
