## Todos los tipos de asincronía

### Categoría 1: Apex Async

| Tipo | Cuándo usarlo |
|---|---|
| `@future` | Callout simple desde trigger, sin necesidad de control |
| `Queueable` | Trabajo complejo, encadenable, con estado |
| `Batch Apex` | Volumen grande de registros (miles / millones) |
| `Scheduled Apex` | Disparar algo en el tiempo (delega a Batch o Queueable, no procesa directo) |

### Categoría 2: Event-Driven

| Tipo | Cuándo usarlo |
|---|---|
| **Platform Events** | Publicar un evento → otro sistema lo consume después. Desacople real |
| **Change Data Capture (CDC)** | Salesforce detecta cambios en registros y publica eventos |

### Categoría 3: UI Async

| Tipo | Cuándo usarlo |
|---|---|
| **Continuation** | Callout HTTP desde LWC/Aura que tarda mucho — libera el hilo del servidor |

### Categoría 4: API Externa Asíncrona

| Tipo | Cuándo usarlo |
|---|---|
| **Bulk API 2.0** | Sistema externo carga/actualiza/borra miles de registros en Salesforce |

### Categoría 5: Automatización Asíncrona

| Tipo | Cuándo usarlo |
|---|---|
| **Scheduled Flow** | Flow que corre en horario programado sin código |
| **Platform Event-Triggered Flow** | Flow que reacciona a un Platform Event |

## @future

El más simple. Método estático anotado que se encola y ejecuta en una transacción separada.

```apex
public class IntegrationService {
    @future(callout=true)
    public static void syncToExternalSystem(Set<Id> contactIds) {
        List<Contact> contacts = [SELECT Id, Email FROM Contact WHERE Id IN :contactIds];
        // aquí ya puedes hacer callouts
    }
}
```

**Reglas críticas:**
- Solo acepta parámetros **primitivos** o colecciones de primitivos — no puedes pasar SObjects
- `callout=true` es obligatorio si vas a hacer HTTP callouts
- Límite: 50 llamadas `@future` por transacción
- No puedes llamar un `@future` desde otro `@future`
- No garantiza orden de ejecución
- No puedes monitorear su estado ni cancelarlo

**Cuándo usarlo:** Operaciones simples de callout desde un trigger. Para casi todo lo demás, Queueable es mejor.

## Queueable

La evolución de `@future`. Implementas la interfaz `Queueable` y tienes mucho más control.

```apex
public class ContactSyncJob implements Queueable, Database.AllowsCallouts {
    private List<Contact> contacts;

    public ContactSyncJob(List<Contact> contacts) {
        this.contacts = contacts;
    }

    public void execute(QueueableContext ctx) {
        for (Contact c : contacts) {
            // procesar, hacer callouts
        }
        if (hayMasTrabajo) {
            System.enqueueJob(new ContactSyncJob(siguienteLote));
        }
    }
}

// Para lanzarlo:
System.enqueueJob(new ContactSyncJob(contacts));
```

**Ventajas sobre @future:**
- Acepta cualquier tipo como parámetro (SObjects, Maps, objetos propios)
- Puedes encadenar — un Queueable puede lanzar otro Queueable
- Tienes el `QueueableContext` con el Job ID para monitorear en `AsyncApexJob`
- Puedes implementar `Database.AllowsCallouts` para HTTP

### Finalizers — manejo de errores robusto

```apex
public class SyncFinalizer implements Finalizer {
    public void execute(FinalizerContext ctx) {
        if (ctx.getResult() == ParentJobResult.UNHANDLED_EXCEPTION) {
            Exception e = ctx.getException();
            // loggear el error, notificar, re-encolar si aplica
        }
    }
}

// Registrar dentro del execute() del Queueable:
System.attachFinalizer(new SyncFinalizer());
```

**Límites:**
- 1 Queueable puede encolar 1 Queueable hijo (en producción)
- En test: `Test.stopTest()` ejecuta el job síncrono, pero no los encadenados

## Batch Apex

Para procesar volúmenes grandes — miles o millones de registros.

```apex
public class ContactMigrationBatch implements Database.Batchable<SObject>, Database.Stateful {

    public Integer procesados = 0; // Database.Stateful preserva esto entre chunks

    public Database.QueryLocator start(Database.BatchableContext bc) {
        return Database.getQueryLocator('SELECT Id, Email FROM Contact WHERE Migrated__c = false');
    }

    public void execute(Database.BatchableContext bc, List<Contact> scope) {
        procesados += scope.size();
        // procesar...
        update scope;
    }

    public void finish(Database.BatchableContext bc) {
        // se llama una vez al final — email de resumen, encolar siguiente batch, etc.
    }
}

// Lanzarlo con scope personalizado:
Database.executeBatch(new ContactMigrationBatch(), 100);
```

### `Database.Stateful`

Sin él, cada chunk `execute()` es una instancia nueva — las variables se resetean. Con él, el estado de la instancia se preserva entre chunks. Úsalo solo cuando lo necesites.

### `QueryLocator` vs `Iterable`

**`QueryLocator`** — es como contratar a una empresa de mensajería:
- Le das la dirección (tu query SOQL)
- Salesforce va, recoge todos los registros y los trae en grupos
- Puede manejar hasta **50 millones de registros**
- Tú no cargas nada — Salesforce maneja la paginación

```apex
public Database.QueryLocator start(Database.BatchableContext bc) {
    return Database.getQueryLocator('SELECT Id, Email FROM Contact WHERE Active__c = true');
}
```

**`Iterable`** — cuando tú ya tienes los datos en tu mesa:
- Tú los recolectaste antes (de una API, de lógica compleja, de un callout previo)
- Le das la lista directamente al Batch
- Solo puedes manejar lo que cabe en memoria (~12MB en async)

```apex
public Iterable<Contact> start(Database.BatchableContext bc) {
    List<Contact> contactsFiltrados = miServicio.obtenerContactosEspeciales();
    return contactsFiltrados;
}
```

**Regla:** Si tus datos vienen de SOQL → `QueryLocator` siempre. Si vienen de lógica que SOQL no puede hacer → `Iterable`.

**Límites:**
- Scope por default: 200 registros por chunk. Máximo: 2,000
- 5 Batch jobs activos simultáneos por org
- Callouts en Batch requieren `Database.AllowsCallouts` y scope=1 si cada registro hace un callout

## Scheduled Apex

Para ejecutar algo en el futuro o en recurrencia.

```apex
public class NightlySync implements Schedulable {
    public void execute(ScheduledContext ctx) {
        Database.executeBatch(new ContactMigrationBatch(), 200);
    }
}

// Programarlo via código:
String cron = '0 0 2 * * ?'; // cada día a las 2am
System.schedule('Nightly Contact Sync', cron, new NightlySync());
```

**CRON en Salesforce:** `Segundos Minutos Horas Día-Mes Mes Día-Semana [Año]`

**Lo que debes saber:**
- Scheduled no procesa directamente — delega a Batch o Queueable
- Máximo 100 jobs programados por org

## Bulk API 2.0

**Para sistemas externos que necesitan cargar/actualizar/borrar muchos datos en Salesforce.**

```
Sistema externo (ERP, Python, MuleSoft...)
    ├─ 1. Crea un "job" en Salesforce vía REST
    ├─ 2. Sube los datos en CSV
    ├─ 3. Cierra el job ("procésalo")
    │       ↓
    │   Salesforce procesa todo en background (asíncrono)
    │       ↓
    └─ 4. El sistema externo consulta el status hasta que termine
         ├─ successfulResults.csv
         └─ failedResults.csv
```

| | Batch Apex | Bulk API 2.0 |
|---|---|---|
| Quién lo usa | Código Apex interno | Sistema externo |
| Cómo se activa | `Database.executeBatch()` | Llamadas REST desde fuera |
| Datos | Registros de Salesforce | CSV que sube el sistema externo |
| Lógica personalizada | Sí, escribes el `execute()` | No, solo operaciones estándar |

## Árbol de decisión: ¿cuál usar?

```
¿Callout desde un trigger?
  └─ @future(callout=true) o Queueable + AllowsCallouts

¿Necesito pasar objetos complejos?
  └─ Queueable (no @future)

¿Más de 10k registros?
  └─ Batch Apex

¿Encadenamiento o manejo de errores robusto?
  └─ Queueable + Finalizer

¿Recurrencia horaria/diaria?
  └─ Scheduled → lanza Batch o Queueable

¿Sistema externo cargando datos masivos?
  └─ Bulk API 2.0

¿Desacople entre sistemas?
  └─ Platform Events
```

## Buenas prácticas de arquitectura

### 1. Tus clases async no tienen lógica de negocio

```apex
// MAL — lógica mezclada con la mecánica async
public void execute(Database.BatchableContext bc, List<Contact> scope) {
    for (Contact c : scope) {
        if (c.Email != null && c.Status__c == 'Active') {
            // 50 líneas de lógica aquí
        }
    }
}

// BIEN — el Batch solo orquesta
public void execute(Database.BatchableContext bc, List<Contact> scope) {
    ContactSyncService.procesarLote(scope);
}
```

### 2. Diseña para idempotencia

Si el job corre dos veces con los mismos datos, el resultado es el mismo. Sin duplicados.

```apex
// Idempotente — usa upsert con External ID
Order__c order = new Order__c(
    External_Id__c = contactId + '_2024Q1', // ID único y predecible
    Contact__c = contactId,
    Amount__c = 100
);
upsert order External_Id__c;
```

### 3. Siempre loggea, nunca tragues excepciones

```apex
try {
    ContactSyncService.procesarLote(scope);
} catch (Exception e) {
    Integration_Log__c log = new Integration_Log__c(
        Job_Id__c        = bc.getJobId(),
        Error__c         = e.getMessage(),
        Stack_Trace__c   = e.getStackTraceString(),
        Records_Affected__c = scope.size()
    );
    insert log;
}
```

### 4. El patrón estándar para jobs complejos

```
Scheduled Apex      → solo lanza, no procesa
    └─ Batch Apex   → procesa el volumen
        └─ finish()
            └─ Queueable     → trabajo post-proceso
                └─ Finalizer → manejo de errores
```

### 5. Platform Events como alternativa a encadenar Apex

En lugar de Queueable→Queueable, publica un Platform Event y deja que otro proceso lo consuma. Desacoplamiento real.

### 6. Monitorea tus jobs siempre

```apex
AsyncApexJob job = [
    SELECT Id, Status, NumberOfErrors, JobItemsProcessed, TotalJobItems
    FROM AsyncApexJob
    WHERE Id = :jobId
];
```

## Ejercicio práctico

Construye un sistema de verificación de emails con estas reglas:

1. Un **Scheduled Apex** dispara cada noche a las 11pm
2. Lanza un **Batch** que procesa Contacts con `Email_Verified__c = false` en chunks de 50
3. Para cada Contact hace un callout simulado a una API de verificación (usa `HttpCalloutMock`)
4. Si el callout falla, un **Finalizer** loggea el error en un objeto `Integration_Error__c`
5. Al terminar el Batch, el `finish()` encola un **Queueable** que envía un email de resumen
6. Publica un **Platform Event** `VerificationCompleted__e` con el resumen final
