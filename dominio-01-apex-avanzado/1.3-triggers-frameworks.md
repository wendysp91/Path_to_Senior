## El orden de ejecución de Salesforce — los 17 pasos

Cuando se guarda un registro:

```
 1. Carga el registro original de la BD (o inicializa si es nuevo)
 2. Carga los nuevos valores del request sobre los viejos
 3. Ejecuta Record-Triggered Flows (before save)
 4. Ejecuta todos los Before Triggers
 5. Ejecuta validation rules (system + custom) y duplicate rules
 6. Guarda en base de datos — NO committed aún, solo en memoria
 7. Ejecuta todos los After Triggers
 8. Ejecuta Assignment Rules
 9. Ejecuta Auto-Response Rules
10. Ejecuta Workflow Rules
    └─ Si workflow actualiza un campo → repite pasos 4, 5, 6, 7 (solo una vez)
11. Ejecuta Process Builder (legacy)
12. Ejecuta Escalation Rules
13. Ejecuta Rollup Summary Fields del padre
14. Ejecuta Criteria-Based Sharing Rules
15. Commit a la base de datos
16. Ejecuta Record-Triggered Flows (after save)
17. Post-commit: envío de emails, encolado de jobs async (@future, Queueable)
```

### Implicaciones clave

| Situación | Qué significa |
|---|---|
| Modificas `Trigger.new` en **before** trigger | Funciona — aún no se guardó |
| Modificas `Trigger.new` en **after** trigger | Error — ya se guardó, es read-only |
| Necesitas el Id del registro nuevo | After trigger — en before insert, el Id no existe |
| Tu `@future` no se ejecuta | Si hubo rollback, el async tampoco corre (paso 17 solo ocurre si paso 15 fue exitoso) |
| Workflow actualiza un campo | Tus before/after triggers corren de nuevo — ¿tu lógica lo maneja? |

## Context Variables — cuáles están disponibles cuándo

```
                    Trigger.new  Trigger.newMap  Trigger.old  Trigger.oldMap
before insert           ✓             ✗               ✗            ✗
before update           ✓             ✓               ✓            ✓
before delete           ✗             ✗               ✓            ✓
after insert            ✓             ✓               ✗            ✗
after update            ✓             ✓               ✓            ✓
after delete            ✗             ✗               ✓            ✓
after undelete          ✓             ✓               ✗            ✗
```

**El que más confunde:** `Trigger.newMap` no existe en before insert — los registros no tienen Id todavía.

### Detectar si un campo cambió (patrón frecuente)

```apex
for (Contact c : Trigger.new) {
    Contact oldContact = Trigger.oldMap.get(c.Id);
    if (c.Email != oldContact.Email) {
        // el email cambió — actuar solo cuando cambia, no en cada update
        c.Email_Verified__c = false;
    }
}
```

## El problema sin framework

Sin framework, el código típicamente luce así:

```apex
trigger ContactTrigger on Contact (before insert, after insert, before update) {
    // 800 líneas mezcladas aquí
}

trigger ContactTrigger2 on Contact (after update) {
    // alguien creó un segundo trigger — orden impredecible
}
```

**Problemas:**
- Dos triggers en el mismo objeto = orden de ejecución no garantizado
- Imposible deshabilitar lógica para migraciones de datos
- No se puede probar de forma aislada

## Patrón base: Un trigger, un handler

**Regla:** Un solo trigger por objeto, que delega todo a una clase handler.

```apex
// ContactTrigger.trigger — solo 10 líneas, nunca más
trigger ContactTrigger on Contact (
    before insert, before update, before delete,
    after insert, after update, after delete, after undelete
) {
    new ContactTriggerHandler().run();
}
```

```apex
// ContactTriggerHandler.cls — aquí va la lógica
public class ContactTriggerHandler {
    public void run() {
        if (Trigger.isBefore) {
            if (Trigger.isInsert) onBeforeInsert(Trigger.new);
            if (Trigger.isUpdate) onBeforeUpdate(Trigger.new, Trigger.oldMap);
            if (Trigger.isDelete) onBeforeDelete(Trigger.old);
        }
        if (Trigger.isAfter) {
            if (Trigger.isInsert)   onAfterInsert(Trigger.new, Trigger.newMap);
            if (Trigger.isUpdate)   onAfterUpdate(Trigger.new, Trigger.newMap, Trigger.oldMap);
            if (Trigger.isDelete)   onAfterDelete(Trigger.oldMap);
            if (Trigger.isUndelete) onAfterUndelete(Trigger.new);
        }
    }

    private void onBeforeInsert(List<Contact> newContacts) {
        ContactService.setDefaultValues(newContacts);
    }

    private void onBeforeUpdate(List<Contact> newContacts, Map<Id, Contact> oldMap) {
        ContactService.handleEmailChange(newContacts, oldMap);
    }

    private void onAfterInsert(List<Contact> newContacts, Map<Id, Contact> newMap) {
        ContactService.syncToExternalSystem(newMap.keySet());
    }
}
```

**El handler no tiene lógica de negocio — solo delega al Service.**

## Kevin O'Hara Trigger Handler — el framework más usado

Agrega tres cosas sobre el patrón base:
1. **Clase base `TriggerHandler`** que todos los handlers extienden
2. **Mecanismo de bypass** — desactivar un handler sin tocar código
3. **Prevención de loops** — detecta si el mismo trigger se ejecuta de nuevo

```apex
// La clase base (se instala una vez, se reutiliza siempre)
public virtual class TriggerHandler {

    private static Set<String> bypassedHandlers = new Set<String>();
    private static Map<String, Integer> loopCountMap = new Map<String, Integer>();

    public void run() {
        if (isBypassed(getHandlerName())) return;
        if (exceedsLoopCount()) return;

        if (Trigger.isBefore) {
            if (Trigger.isInsert)   beforeInsert();
            if (Trigger.isUpdate)   beforeUpdate();
            if (Trigger.isDelete)   beforeDelete();
        }
        if (Trigger.isAfter) {
            if (Trigger.isInsert)   afterInsert();
            if (Trigger.isUpdate)   afterUpdate();
            if (Trigger.isDelete)   afterDelete();
            if (Trigger.isUndelete) afterUndelete();
        }
    }

    // Métodos virtuales — el handler hijo hace override solo de los que necesita
    protected virtual void beforeInsert()  {}
    protected virtual void beforeUpdate()  {}
    protected virtual void beforeDelete()  {}
    protected virtual void afterInsert()   {}
    protected virtual void afterUpdate()   {}
    protected virtual void afterDelete()   {}
    protected virtual void afterUndelete() {}

    // Bypass
    public static void bypass(String handlerName) {
        bypassedHandlers.add(handlerName);
    }
    public static void clearBypass(String handlerName) {
        bypassedHandlers.remove(handlerName);
    }
    public static Boolean isBypassed(String handlerName) {
        return bypassedHandlers.contains(handlerName);
    }

    private String getHandlerName() {
        return String.valueOf(this).substring(0, String.valueOf(this).indexOf(':'));
    }

    private Boolean exceedsLoopCount() {
        String name = getHandlerName();
        if (!loopCountMap.containsKey(name)) loopCountMap.put(name, 0);
        Integer count = loopCountMap.get(name);
        if (count >= 5) return true;
        loopCountMap.put(name, count + 1);
        return false;
    }
}
```

```apex
// Tu handler — limpio, solo override lo que necesitas
public class ContactTriggerHandler extends TriggerHandler {

    private List<Contact> newContacts;
    private Map<Id, Contact> oldMap;

    public ContactTriggerHandler() {
        this.newContacts = Trigger.new;
        this.oldMap = (Map<Id, Contact>) Trigger.oldMap;
    }

    protected override void beforeInsert() {
        ContactService.setDefaultValues(newContacts);
    }

    protected override void afterUpdate() {
        ContactService.handleEmailChange(newContacts, oldMap);
    }
}
```

### El bypass en acción

```apex
// En una migración de datos
TriggerHandler.bypass('ContactTriggerHandler');
insert contactsMigrados;
TriggerHandler.clearBypass('ContactTriggerHandler');

// En un test, aislar comportamiento
TriggerHandler.bypass('AccountTriggerHandler');
insert contactsDeTest;
TriggerHandler.clearBypass('AccountTriggerHandler');
```

## Custom Metadata-Driven — el siguiente nivel

Para orgs grandes donde quieres controlar handlers sin deploy:

```
Trigger_Setting__mdt
├── Handler_Class__c = 'ContactEmailHandler'
├── Is_Active__c = true
├── Order__c = 10
└── SObject__c = 'Contact'
```

```apex
trigger ContactTrigger on Contact (before insert, after insert, before update, after update) {
    List<Trigger_Setting__mdt> settings = [
        SELECT Handler_Class__c, Order__c
        FROM Trigger_Setting__mdt
        WHERE SObject__c = 'Contact' AND Is_Active__c = true
        ORDER BY Order__c ASC
    ];

    for (Trigger_Setting__mdt setting : settings) {
        Type handlerType = Type.forName(setting.Handler_Class__c);
        TriggerHandler handler = (TriggerHandler) handlerType.newInstance();
        handler.run();
    }
}
```

**Ventaja:** Activas/desactivas handlers desde Setup > Custom Metadata — sin deploy.

## Cuándo usar cada approach

```
¿Proyecto pequeño, equipo de 1-2?
  └─ Patrón base: un trigger + un handler

¿Equipo mediano, necesitas bypass para migraciones?
  └─ Kevin O'Hara TriggerHandler

¿Proyecto enterprise, múltiples equipos?
  └─ Custom Metadata-driven

¿Trabajas con NPSP?
  └─ TDTM (viene en NPSP, aprende a extenderlo)

¿Proyecto con fflib/Apex Enterprise Patterns?
  └─ fflib Domain Layer (ver Dominio 8)
```

## Ejercicio práctico

Tienes este trigger en producción que causa problemas en migraciones. Refactorízalo:

```apex
trigger OpportunityTrigger on Opportunity (before insert, before update, after insert, after update) {
    for (Opportunity opp : Trigger.new) {
        if (Trigger.isInsert) {
            opp.Stage__c = 'New';
            opp.Assigned_Date__c = Date.today();
        }
        if (Trigger.isUpdate) {
            if (opp.StageName == 'Closed Won') {
                Opportunity old = Trigger.oldMap.get(opp.Id);
                if (old.StageName != 'Closed Won') {
                    // notificar equipo
                }
            }
        }
    }
    if (Trigger.isAfter && Trigger.isInsert) {
        OpportunityService.syncToERP(Trigger.new);
    }
}
```

**Tu tarea:**
1. Identifica todos los problemas de diseño
2. Refactoriza usando el patrón Kevin O'Hara
3. Agrega bypass para que la migración pueda correr sin disparar `syncToERP`
4. Asegúrate de que "detectar cambio de Stage a Closed Won" sea bulk-safe
