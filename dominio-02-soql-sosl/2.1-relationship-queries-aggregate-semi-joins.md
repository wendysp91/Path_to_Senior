## Relationship Queries

### Parent-to-Child (Inner Query / Subquery)
Desde un objeto padre, traer sus registros hijo en el mismo query.

```soql
SELECT Id, Name,
       (SELECT Id, LastName, Email FROM Contacts)
FROM Account
WHERE Industry = 'Technology'
```

- El objeto hijo va en el `FROM` de la subquery con su **nombre de relaciĂłn** (plural), no el API name del objeto.
- El resultado padre tiene una lista de sObjects hijo en el campo de relaciĂłn.
- LĂ­mite: hasta **1 nivel** de subquery anidada.

```apex
List<Account> accounts = [
    SELECT Id, Name, (SELECT Id, LastName FROM Contacts)
    FROM Account
];

for (Account acc : accounts) {
    for (Contact c : acc.Contacts) {
        System.debug(c.LastName);
    }
}
```

### Child-to-Parent (Dot Notation)
Desde un objeto hijo, navegar hacia campos del padre.

```soql
SELECT Id, LastName, Account.Name, Account.Industry
FROM Contact
WHERE Account.BillingCity = 'Mexico City'
```

- Se usa notaciĂłn de punto: `RelationshipName.Field`.
- Para relaciones estĂˇndar: `Account.Name`, `Owner.Email`, etc.
- Para relaciones custom: el lookup field termina en `__r` â†’ `MyObject__r.Name`.
- Puedes navegar hasta **5 niveles** de profundidad.

```soql
-- EstĂˇndar

-- Custom
SELECT Name, Custom_Lookup__r.Name FROM MyObject__c
```

### Nombres de relaciĂłn â€” EstĂˇndar vs Custom

| Tipo | Lookup Field | Nombre de relaciĂłn (childâ†’parent) | Nombre de relaciĂłn (parentâ†’child) |

## Aggregate Functions

Se usan en `SELECT` junto con `GROUP BY`. Devuelven `List<AggregateResult>`, no sObjects.

| FunciĂłn | DescripciĂłn |
|---|---|
| `COUNT()` | Cuenta filas (sin campo) |
| `COUNT(field)` | Cuenta valores no nulos |
| `COUNT_DISTINCT(field)` | Cuenta valores Ăşnicos |
| `SUM(field)` | Suma |
| `AVG(field)` | Promedio |
| `MIN(field)` | MĂ­nimo |
| `MAX(field)` | MĂˇximo |

```soql
SELECT AccountId, COUNT(Id), SUM(Amount), AVG(Amount)
FROM Opportunity
WHERE StageName = 'Closed Won'
GROUP BY AccountId
```

### Leer AggregateResult en Apex

```apex
List<AggregateResult> results = [
    SELECT AccountId, COUNT(Id) total, SUM(Amount) revenue
    FROM Opportunity
    WHERE StageName = 'Closed Won'
    GROUP BY AccountId
];

for (AggregateResult ar : results) {
    Id accId    = (Id)      ar.get('AccountId');
    Integer cnt = (Integer) ar.get('total');
    Decimal rev = (Decimal) ar.get('revenue');
}
```

> Cuando usas alias (`COUNT(Id) total`) lo accedes con `ar.get('total')`.  
> Si no pones alias, el nombre generado es `expr0`, `expr1`, etc.

### HAVING â€” filtrar sobre agregados

`HAVING` filtra el resultado del `GROUP BY`, igual que `WHERE` pero aplicado despuĂ©s de agrupar.

```soql
SELECT AccountId, COUNT(Id) total
FROM Opportunity
GROUP BY AccountId
HAVING COUNT(Id) > 5
```

No puedes mezclar campos normales con aggregate functions en `SELECT` sin incluirlos en `GROUP BY`.

### GROUP BY ROLLUP / CUBE

Para subtotales:

```soql
-- ROLLUP: subtotales por nivel
SELECT Type, LeadSource, COUNT(Id)
FROM Lead
GROUP BY ROLLUP(Type, LeadSource)

-- CUBE: todas las combinaciones de subtotales
SELECT Type, LeadSource, COUNT(Id)
FROM Lead
GROUP BY CUBE(Type, LeadSource)
```

Los grupos con subtotal tienen `null` en el campo agrupado donde se calculĂł el subtotal.

## Semi-Joins y Anti-Joins

Permiten filtrar un query basado en la existencia (o ausencia) de registros en otro objeto, sin hacer dos queries.

### Semi-Join (IN + subquery)

"Dame los registros donde existe una coincidencia en el subquery."

```soql
-- Contacts cuya Account tiene Opportunities abiertas
SELECT Id, LastName
FROM Contact
WHERE AccountId IN (
    SELECT AccountId
    FROM Opportunity
    WHERE StageName != 'Closed Won'
)
```

### Anti-Join (NOT IN + subquery)

"Dame los registros donde NO existe coincidencia."

```soql
-- Accounts sin ningĂşn Contact
SELECT Id, Name
FROM Account
WHERE Id NOT IN (
    SELECT AccountId
    FROM Contact
    WHERE AccountId != null
)
```

### Reglas clave de semi/anti-joins

- El campo del `WHERE` exterior y el campo del `SELECT` interior deben ser el **mismo tipo de relaciĂłn** (Id del mismo objeto o lookup al mismo objeto).
- La subquery solo puede tener **un campo en su SELECT**.
- No puedes anidar mĂˇs subqueries dentro del semi-join.
- SĂ­ puedes combinar: `IN (subquery)` y `NOT IN (subquery)` en el mismo query.
- Se pueden usar en subqueries parent-to-child tambiĂ©n:

```soql
-- Accounts que tienen Contacts con tĂ­tulo 'CEO'
SELECT Id, Name,
       (SELECT Id, LastName FROM Contacts WHERE Title = 'CEO')
FROM Account
WHERE Id IN (
    SELECT AccountId FROM Contact WHERE Title = 'CEO'
)
```
