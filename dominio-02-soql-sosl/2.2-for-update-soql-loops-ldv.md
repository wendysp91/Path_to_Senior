# 2.2 — FOR UPDATE, SOQL For Loops, Large Data Volumes

---

## FOR UPDATE

Bloquea registros a nivel de base de datos para evitar condiciones de carrera cuando múltiples transacciones leen y modifican los mismos registros.

```soql
SELECT Id, Status__c FROM Invoice__c WHERE Status__c = 'Pending' FOR UPDATE
```

### Cómo funciona

- Los registros quedan **bloqueados** durante toda la transacción hasta que hace commit o rollback.
- Otros procesos que intenten hacer `FOR UPDATE` sobre los mismos registros **esperan** hasta que el bloqueo se libere.
- Si el bloqueo no se libera a tiempo, la otra transacción recibe un error de lock.

### Cuándo usarlo

- Procesos batch o async que compiten por los mismos registros.
- Cuando haces read → modify → write y no puedes permitir que otro proceso modifique el registro entre el read y el write.

```apex
List<Account> accounts = [SELECT Id, Balance__c FROM Account WHERE Id = :targetId FOR UPDATE];
// Nadie más puede modificar estos registros hasta que termine esta transacción
accounts[0].Balance__c -= 500;
update accounts;
```

### Restricciones

- No puedes usar `FOR UPDATE` con `ORDER BY`, `LIMIT`, ni aggregate functions.
- No funciona en queries async (Batch execute, @future). Sí funciona en Queueable.
- No aplica a registros de solo lectura ni a External Objects.

## SOQL For Loops

Permiten iterar sobre resultados de un query sin cargar todos los registros en memoria al mismo tiempo. Esenciales para manejar grandes volúmenes de datos sin violar el heap size limit (6 MB sincrono / 12 MB async).

### Sintaxis — Registro por registro

```apex
for (Account acc : [SELECT Id, Name FROM Account WHERE IsActive__c = true]) {
    // Se procesa un registro a la vez
    // El heap solo mantiene el registro actual
}
```

### Sintaxis — Por lotes de 200 (la más importante)

```apex
for (List<Account> batch : [SELECT Id, Name FROM Account WHERE IsActive__c = true]) {
    // batch tiene hasta 200 registros por iteración
    update batch; // DML eficiente: 1 llamada por cada 200 registros
}
```

### Por qué la variante de lista es mejor

| Aspecto | Registro por registro | Lista (lotes de 200) |
|---|---|---|
| Heap | Mínimo | Bajo |
| DML dentro del loop | 1 DML por registro ❌ | 1 DML por 200 registros ✅ |
| Governor limits DML | Se agota rápido | Eficiente |
| Uso recomendado | Lectura simple | Modificación masiva |

### Regla crítica

Nunca hagas DML dentro de un loop de registro por registro. Siempre acumula en una lista y ejecuta el DML fuera del loop, o usa la variante de lista de 200.

```apex
// MAL — 1 DML por registro, revienta governor limits
for (Account acc : [SELECT Id FROM Account]) {
    acc.Rating = 'Hot';
    update acc; // ❌
}

// BIEN — 1 DML total
List<Account> toUpdate = new List<Account>();
for (Account acc : [SELECT Id FROM Account]) {
    acc.Rating = 'Hot';
    toUpdate.add(acc);
}
update toUpdate; // ✅

// TAMBIÉN BIEN — 1 DML por cada 200
for (List<Account> batch : [SELECT Id FROM Account]) {
    for (Account acc : batch) acc.Rating = 'Hot';
    update batch; // ✅
}
```

## Large Data Volumes (LDV)

Consideraciones cuando un objeto tiene millones de registros.

### Síntomas de LDV

- Queries lentas o que agotan el timeout de 120 segundos.
- Lock contention en DML masivo.
- Reportes y list views que se cuelgan.

### Indexación en Salesforce

Salesforce indexa automáticamente:

| Campo | Indexado automáticamente |
|---|---|
| `Id` | ✅ Siempre |
| `Name` | ✅ |
| `CreatedDate` | ✅ |
| `SystemModstamp` | ✅ |
| Campos de lookup/master-detail | ✅ |
| `RecordTypeId` | ✅ |
| Campos marcados como **External ID** | ✅ |
| Campos marcados como **Unique** | ✅ |

Campos como `Email`, `Phone`, campos de texto libre → **NO indexados por defecto**.

### Selectividad

Un query es **selectivo** cuando el filtro reduce el resultado a menos del umbral:

| Volumen del objeto | Umbral de selectividad |
|---|---|
| < 1 millón de registros | 10% de los registros |
| > 1 millón de registros | 5% de los registros o 800,000 registros (el menor) |

Si el query no es selectivo y el objeto es LDV, Salesforce puede rechazarlo o ejecutarlo con full table scan (muy lento).

### Cómo garantizar selectividad

1. **Filtrar siempre por campos indexados** (`Id`, `CreatedDate`, External ID, lookups).
2. **Evitar filtros con funciones** — `WHERE CALENDAR_YEAR(CreatedDate) = 2025` no usa el índice.
3. **Evitar leading wildcards** — `WHERE Name LIKE '%Corp'` no usa el índice.
4. **Usar fechas relativas con precaución** — `LAST_N_DAYS:30` sí aprovecha el índice de `CreatedDate`.

### Custom Indexes

Para campos que no están indexados y son usados frecuentemente en WHERE, puedes solicitar a Salesforce Support un **Custom Index**. No se pueden crear desde Setup.

Candidatos para custom index:
- Campo de texto usado como filtro principal en LDV objects.
- Campo de estado/categoría con alta cardinalidad (muchos valores distintos).

> Custom index NO ayuda si el campo tiene muchos nulos o pocos valores distintos (baja cardinalidad). El optimizador puede decidir ignorarlo.

### Skinny Tables

Tabla interna que Salesforce puede crear (solo vía Support) con un subconjunto de campos de un objeto LDV para acelerar queries específicos. No es configurable por el desarrollador.

### División de datos con archivado

Para objetos muy grandes, una estrategia es archivar registros históricos usando:
- **Big Objects** — almacenamiento de solo escritura, sin límite práctico, accesible vía SOQL restringido o Async SOQL.
- **External Objects** — datos en sistema externo, accesibles como si fueran un objeto Salesforce.

```soql
-- Big Object: solo soporta filtros por campos del índice compuesto, en orden
SELECT Id, EventDate__c, EventType__c
FROM MyBigObject__b
WHERE EventDate__c >= 2024-01-01T00:00:00Z
```

### Async SOQL

Para queries muy pesados en LDV que no caben en la transacción síncrona. Ejecuta el query en background y escribe los resultados en un Big Object o sObject.

No es código Apex — se invoca vía REST API de Tooling o desde Bulk API. No disponible en Developer Edition.

## Resumen rápido

| Concepto | Clave |
|---|---|
| `FOR UPDATE` | Bloquea registros para evitar race conditions; no compatible con ORDER BY ni async |
| SOQL for loop (registro) | Bajo heap, pero no hagas DML dentro |
| SOQL for loop (lista 200) | El patrón correcto para DML masivo en Apex |
| LDV selectividad | < 10% (o 5% si >1M) de filas para que el query use el índice |
| Custom Index | Pedirlo a Support para campos no indexados usados en WHERE |
| Big Objects | Archivado sin límite, SOQL restringido al índice compuesto |
