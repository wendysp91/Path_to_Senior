# 2.3 — Query Plan Tool, Selectividad e Índices

---

## Query Plan Tool

Herramienta en Developer Console que muestra cómo el optimizador de Salesforce planea ejecutar un SOQL query, sin ejecutarlo realmente.

### Cómo activarlo

1. Developer Console → Help → Preferences → `Enable Query Plan` → ✅
2. En la pestaña **Query Editor**, escribe tu SOQL y haz clic en **Query Plan** (en lugar de Execute).

### Qué muestra

| Columna | Significado |
|---|---|
| **Cardinality** | Número estimado de filas que retornará el plan |
| **Fields** | Campos usados para el plan (índices aprovechados) |
| **Leading Operation Type** | Tipo de operación principal (ver abajo) |
| **Cost** | Costo relativo del plan — **menor es mejor** |
| **SObject Cardinality** | Total de registros en el objeto |

### Leading Operation Types

| Tipo | Qué significa |
|---|---|
| `Index` | Usa un índice → rápido ✅ |
| `TableScan` | Escanea toda la tabla → lento en LDV ❌ |
| `Other` | Operación especial (ej. compound filter) |

### Cómo interpretar el costo

- **Costo < 1** → el query es selectivo, usa índice.
- **Costo = 1** → equivale a un TableScan completo.
- **Costo > 1** → peor que un TableScan (raro, indica query muy malo).

El optimizador **elige el plan de menor costo**. Si todos los planes tienen costo cercano a 1, hará TableScan.

```
Ejemplo de salida:
┌─────────────┬──────────────────┬──────────────────────┬──────┐
│ Cardinality │ Fields           │ Leading Op Type      │ Cost │
├─────────────┼──────────────────┼──────────────────────┼──────┤
│ 150         │ CreatedDate      │ Index                │ 0.03 │
│ 50,000      │ —                │ TableScan            │ 1.0  │
└─────────────┴──────────────────┴──────────────────────┴──────┘
→ El optimizador elige el plan con Index (costo 0.03)
```

### Limitaciones del Query Plan Tool

- Solo disponible en Developer Console, no en VS Code ni Workbench.
- Muestra estimados basados en estadísticas, no ejecución real.
- No aplica a SOSL.
- No considera el contexto de sharing (with/without sharing).

## Selectividad

Un query es selectivo cuando el filtro es lo suficientemente restrictivo para que Salesforce use el índice en lugar de un TableScan.

### Umbrales de selectividad

| Registros en el objeto | Umbral |
|---|---|
| Cualquier tamaño | < 10% de los registros totales |
| > 1,000,000 registros | < 5% **o** < 800,000 registros (el menor de los dos) |

### Cómo mejora la selectividad

**Filtros que SÍ usan índice:**
```soql
WHERE Id = '001...'                        -- siempre selectivo
WHERE CreatedDate >= 2025-01-01T00:00:00Z  -- índice en fecha
WHERE AccountId = :accId                   -- lookup indexado
WHERE ExternalId__c = 'EXT-001'            -- External ID indexado
WHERE IsActive__c = true AND CreatedDate > LAST_YEAR  -- compound
```

**Filtros que NO usan índice:**
```soql
WHERE Name LIKE '%Corp'         -- leading wildcard
WHERE CALENDAR_YEAR(CreatedDate) = 2025  -- función sobre campo indexado
WHERE Description__c = 'algo'  -- campo de texto largo (textarea)
WHERE Phone = '5512345678'      -- campo no indexado por defecto
WHERE IsActive__c = false       -- campos con mayoría de valores null o false
                                -- pueden considerarse no selectivos
```

### Compound filters (AND / OR)

- Con `AND`: basta con que **un** filtro sea selectivo → el optimizador puede usar ese índice.
- Con `OR`: **todos** los filtros deben ser selectivos para que el query lo sea.

```soql
-- AND: si CreatedDate es selectivo, el query es selectivo aunque Status no lo sea
WHERE Status__c = 'Active' AND CreatedDate >= LAST_YEAR

-- OR: ambos deben ser selectivos
WHERE CreatedDate >= LAST_YEAR OR ExternalId__c = 'EXT-001'
```

## Índices en Salesforce

### Índices estándar (automáticos)

Salesforce crea y mantiene índices automáticos en:

- `Id`
- `Name`
- `OwnerId`
- `CreatedDate`, `LastModifiedDate`, `SystemModstamp`
- Campos de lookup y master-detail (`AccountId`, `ContactId`, etc.)
- `RecordTypeId`
- Campos marcados como **External ID**
- Campos marcados como **Unique**
- `IsDeleted` (para la papelera)

### Índices custom

Para campos no incluidos arriba. Solo se pueden solicitar a **Salesforce Support** (caso en Help Portal). No hay opción en Setup para crearlos.

**Cuándo pedirlo:**
- Campo usado frecuentemente en `WHERE` de queries sobre objetos LDV.
- El Query Plan Tool muestra TableScan para ese campo.
- La selectividad del campo es buena (muchos valores distintos, pocos nulos).

**Cuándo NO ayuda un índice custom:**
- Campo con baja cardinalidad (ej. un campo Picklist con 3 valores en 5M registros).
- Campo con muchos valores nulos (el índice de Salesforce no indexa nulos en la mayoría de casos).
- El query ya es un TableScan por otro motivo (OR con campo no indexado).

### Two-Column Index (índice compuesto)

Salesforce soporta índices de dos columnas. Se solicitan también a Support. Útiles cuando el filtro siempre combina dos campos:

```soql
-- Siempre filtras por Status + Region → solicitar índice compuesto (Status, Region)
WHERE Status__c = 'Active' AND Region__c = 'North'
```

### Índices en Big Objects

Big Objects tienen un **índice compuesto obligatorio** definido en su metadata. El SOQL sobre Big Objects **solo puede filtrar por los campos del índice y en el orden definido**:

```xml
<!-- Definición del índice compuesto en el Big Object -->
<indexes>
    <fullName>EventDate__c</fullName>
    <sortDirection>DESC</sortDirection>
</indexes>
<indexes>
    <fullName>EventType__c</fullName>
    <sortDirection>ASC</sortDirection>
</indexes>
```

```soql
-- Válido: filtra por el primer campo del índice
SELECT EventDate__c, EventType__c FROM MyEvent__b WHERE EventDate__c >= 2024-01-01T00:00:00Z

-- Inválido: no puedes saltarte el primer campo
SELECT EventDate__c FROM MyEvent__b WHERE EventType__c = 'Login'
```

## Flujo de diagnóstico para queries lentos

```
Query lento o timeout
        │
        ▼
Abrir Query Plan Tool
        │
        ├── Leading Op: Index, Cost < 0.1 ──→ El query está bien, problema puede ser volumen de resultado
        │
        └── Leading Op: TableScan, Cost ≈ 1
                │
                ├── ¿El campo WHERE está indexado?
                │       No ──→ Solicitar Custom Index a Support
                │       Sí ──→ ¿El filtro tiene baja cardinalidad o muchos nulos?
                │                       Sí ──→ Reestructurar query (añadir campo indexado con AND)
                │
                ├── ¿Hay OR con campo no indexado?
                │       Sí ──→ Dividir en dos queries + combinar en Apex
                │
                └── ¿Hay función sobre el campo? (CALENDAR_YEAR, etc.)
                        Sí ──→ Reemplazar con rango de fechas explícito
```

### Reemplazar función por rango equivalente

```soql
-- MAL: no usa índice
WHERE CALENDAR_YEAR(CreatedDate) = 2025

-- BIEN: usa índice de CreatedDate
WHERE CreatedDate >= 2025-01-01T00:00:00Z AND CreatedDate < 2026-01-01T00:00:00Z
```

## Resumen rápido

| Concepto | Clave |
|---|---|
| Query Plan Tool | Developer Console → Query Editor → Query Plan. Cost < 1 = usa índice |
| `Index` vs `TableScan` | Index = rápido, TableScan = lento en LDV |
| Selectividad | < 10% filas (< 5% si >1M) para que use índice |
| AND vs OR | AND: un filtro selectivo basta. OR: todos deben ser selectivos |
| Custom Index | Solo vía Salesforce Support. Útil con alta cardinalidad y pocos nulos |
| Leading wildcard | `LIKE '%texto'` rompe el índice |
| Función en WHERE | `CALENDAR_YEAR(...)` rompe el índice → usar rango explícito |
| Big Objects | SOQL restringido al orden del índice compuesto definido en metadata |

## Preguntas de práctica

1. ¿Qué significa que el `Cost` del Query Plan sea 1.0?
2. Tienes un query con `OR` entre dos campos. ¿Qué condición se necesita para que el query sea selectivo?
3. ¿Cómo reescribes `WHERE FISCAL_MONTH(CloseDate) = 3` para que use el índice?
4. Un campo `Status__c` (picklist: Active/Inactive) en un objeto de 3M registros donde el 95% son 'Active'. Pides un custom index. ¿Ayudará? ¿Por qué?
5. En un Big Object con índice compuesto `(Region__c, EventDate__c)`, ¿es válido este query? `WHERE EventDate__c >= 2025-01-01T00:00:00Z`
