# 4.2 — Callouts: Named Credentials, OAuth 2.0, JWT

---

## Callouts en Apex

Apex puede hacer llamadas HTTP a sistemas externos usando `HttpRequest` / `Http`.

```apex
HttpRequest req = new HttpRequest();
req.setEndpoint('https://api.ejemplo.com/data');
req.setMethod('GET');
req.setHeader('Authorization', 'Bearer ' + token);
req.setTimeout(10000); // ms, máx 120,000

Http http = new Http();
HttpResponse res = http.send(req);

if (res.getStatusCode() == 200) {
    Map<String, Object> body = (Map<String, Object>) JSON.deserializeUntyped(res.getBody());
}
```

### Restricciones de callouts

| Límite | Valor |
|---|---|
| Callouts por transacción | 100 |
| Timeout máximo por callout | 120,000 ms (120 s) |
| Tamaño máximo de respuesta | 6 MB |
| Callouts sincrónicos | No puedes hacer DML antes de un callout en la misma transacción |
| Callouts en trigger | Solo en contexto async (`@future(callout=true)`, Queueable, etc.) |

### DML antes de callout

```apex
// MAL — DML antes de callout en la misma transacción
insert myRecord;
Http http = new Http(); // ❌ System.CalloutException

// BIEN — separar en contexto async
insert myRecord;
MyQueueable.enqueue(myRecord.Id); // el callout va en el Queueable
```

## Named Credentials

Almacenan la URL del endpoint y las credenciales (token, usuario/contraseña, certificado) de forma segura en Salesforce. El desarrollador no toca credenciales directamente en el código.

### Ventajas

- Las credenciales no están en el código ni en Custom Settings.
- Salesforce gestiona el refresh del token automáticamente (para OAuth).
- Se puede cambiar la URL/credencial sin modificar código.
- Permite controlar acceso con permisos de perfil.

### Sintaxis en Apex

```apex
HttpRequest req = new HttpRequest();
req.setEndpoint('callout:MiNamedCredential/api/v1/orders');
req.setMethod('GET');
// Sin header Authorization — Named Credential lo agrega automáticamente
Http http = new Http();
HttpResponse res = http.send(req);
```

El prefijo `callout:` le dice a Salesforce que use la Named Credential con ese nombre.

### Named Credential vs External Credential (modelo nuevo)

Desde Summer '23, Salesforce separó el concepto en dos:

| Componente | Qué guarda |
|---|---|
| **External Credential** | El protocolo de autenticación (OAuth, JWT, Custom Header, etc.) y los parámetros (client_id, secret, etc.) |
| **Named Credential** | La URL del endpoint, vinculada a un External Credential |
| **Principal** | La identidad dentro del External Credential (org-wide o por usuario) |

El modelo legacy (Named Credential con todo junto) sigue funcionando pero el nuevo modelo es más flexible y seguro.

## OAuth 2.0 — Grant Types clave

### Authorization Code Flow (3-legged)

El usuario autoriza manualmente. Usado para integraciones donde cada usuario tiene su propia identidad en el sistema externo.

```
1. Salesforce redirige al usuario a la URL de autorización del proveedor
2. Usuario hace login y aprueba
3. Proveedor devuelve un authorization_code
4. Salesforce intercambia el code por access_token + refresh_token
5. Salesforce almacena el refresh_token y renueva el access_token automáticamente
```

### Client Credentials Flow (2-legged)

Sin intervención del usuario. Usado para integraciones sistema-a-sistema donde no hay identidad de usuario.

```
POST /token
  grant_type=client_credentials
  &client_id={client_id}
  &client_secret={client_secret}

→ Respuesta: { "access_token": "...", "token_type": "Bearer", "expires_in": 3600 }
```

### Username-Password Flow (legacy, no recomendado)

```
POST /token
  grant_type=password
  &username={user}
  &password={password+security_token}
  &client_id={id}
  &client_secret={secret}
```

> Desaconsejado porque expone credenciales de usuario. Salesforce lo deshabilitará para nuevos orgs.

### Refresh Token Flow

Cuando el access_token expira, se usa el refresh_token para obtener uno nuevo sin re-autenticar:

```
POST /token
  grant_type=refresh_token
  &refresh_token={refresh_token}
  &client_id={id}
  &client_secret={secret}
```

## JWT Bearer Flow

Permite autenticación sin secretos compartidos. El cliente firma un JWT con su clave privada; Salesforce (u otro proveedor) verifica con la clave pública.

### Flujo completo

```
1. Generar un par de claves (privada + pública) y crear un Certificado en Salesforce
2. Crear una Connected App con el certificado público
3. El cliente construye un JWT con claims específicos
4. El cliente firma el JWT con su clave privada
5. POST al token endpoint con el JWT firmado
6. Salesforce verifica la firma con la clave pública del certificado
7. Salesforce devuelve un access_token
```

### Estructura del JWT

```
Header.Payload.Signature

Header:  { "alg": "RS256", "typ": "JWT" }
Payload: {
    "iss": "{client_id de la Connected App}",
    "sub": "{username del usuario en Salesforce}",
    "aud": "https://login.salesforce.com",
    "exp": {timestamp unix + 3 min}
}
```

### En Apex — JWT Bearer para llamar a un sistema externo

```apex
// 1. Crear el header y payload
String header  = JSON.serialize(new Map<String,String>{'alg'=>'RS256','typ'=>'JWT'});
String payload = JSON.serialize(new Map<String,Object>{
    'iss' => clientId,
    'sub' => subject,
    'aud' => tokenEndpoint,
    'exp' => (DateTime.now().addMinutes(3)).getTime() / 1000
});

String headerB64  = EncodingUtil.base64Encode(Blob.valueOf(header));
String payloadB64 = EncodingUtil.base64Encode(Blob.valueOf(payload));
String signingInput = headerB64 + '.' + payloadB64;

// 2. Firmar con clave privada (certificado en Salesforce)
Blob signature = Crypto.signWithCertificate('RSA-SHA256', Blob.valueOf(signingInput), 'MiCertificado');
String signatureB64 = EncodingUtil.base64Encode(signature);

String jwt = signingInput + '.' + signatureB64;

// 3. Intercambiar JWT por access_token
HttpRequest req = new HttpRequest();
req.setEndpoint(tokenEndpoint);
req.setMethod('POST');
req.setBody('grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer&assertion=' + jwt);
HttpResponse res = new Http().send(req);
```

### Named Credential con JWT

La forma más limpia: configurar la Named Credential con External Credential de tipo JWT, y Salesforce gestiona todo el proceso automáticamente sin código de firma manual.

## Certificados en Salesforce

Dos tipos:

| Tipo | Uso |
|---|---|
| **Self-Signed Certificate** | Generado por Salesforce. La clave privada nunca sale de Salesforce. Ideal para JWT Bearer. |
| **CA-Signed Certificate** | Firmado por una autoridad certificadora. Para mTLS u otros protocolos que requieren validación externa. |

Ubicación: Setup → Certificate and Key Management.

```apex
// Firmar con un certificado de Salesforce (clave privada en Salesforce)
Blob signature = Crypto.signWithCertificate('RSA-SHA256', dataBlob, 'NombreCertificado');

// Verificar firma con clave pública
Boolean valid = Crypto.verify('RSA-SHA256', dataBlob, signatureBlob, publicKeyBlob);
```

## Resumen rápido

| Concepto | Clave |
|---|---|
| Callout en trigger | Solo en async (`@future(callout=true)`, Queueable) |
| DML antes de callout | Rompe la transacción — separar en async |
| Named Credential | `callout:NombreNC/path` — Salesforce maneja las credenciales |
| OAuth Authorization Code | 3-legged — el usuario autoriza manualmente |
| OAuth Client Credentials | 2-legged — sistema a sistema, sin usuario |
| JWT Bearer | Sin secreto compartido — firma con clave privada, verificación con pública |
| `Crypto.signWithCertificate` | Firma con clave privada almacenada en Salesforce |
| External Credential (nuevo) | Separa protocolo de autenticación del endpoint |
