# 4.4 — Event-Driven Architecture: Platform Events y Change Data Capture

---

## Event-Driven Architecture (EDA)

Arquitectura donde los sistemas se comunican mediante eventos en lugar de llamadas directas. Los productores publican eventos sin saber quién los consume; los consumidores se suscriben sin conocer al productor.

**Beneficios:**
- Desacoplamiento entre sistemas.
- Escalabilidad — múltiples consumidores sin cambiar el productor.
- Resiliencia — si el consumidor cae, el evento queda en el bus hasta que pueda procesarlo.

---

## Platform Events

Mensajes de datos custom que fluyen a través del bus de eventos de Salesforce. Pueden ser publicados y consumidos desde Apex, Flow, procesos externos vía API y LWC.

### Definición (metadata)

```xml
<!-- force-app/main/default/objects/OrderShipped__e/OrderShipped__e.object-meta.xml -->
<CustomObject>
    <label>Order Shipped</label>
    <pluralLabel>Order Shipped Events</pluralLabel>
    <deploymentStatus>Deployed</deploymentStatus>
    <eventType>StandardVolume</eventType>  <!-- o HighVolume -->
    <fields>
        <fullName>OrderId__c</fullName>
        <type>Text</type>
        <length>18</length>
    </fields>
    <fields>
        <fullName>Status__c</fullName>
        <type>Text</type>
        <length>50</length>
    </fields>
</CustomObject>
```

### Publicar desde Apex

```apex
// Publicar un evento
OrderShipped__e event = new OrderShipped__e(
    OrderId__c = order.Id,
    Status__c  = 'Shipped'
);
Database.SaveResult result = EventBus.publish(event);

if (!result.isSuccess()) {
    System.debug('Error: ' + result.getErrors()[0].getMessage());
}
```

Para publicar en lote:
```apex
List<OrderShipped__e> events = new List<OrderShipped__e>{ ev1, ev2, ev3 };
List<Database.SaveResult> results = EventBus.publish(events);
```

### Publicar desde API externa

```http
POST /services/data/v59.0/sobjects/OrderShipped__e
Authorization: Bearer {token}
Content-Type: application/json

{ "OrderId__c": "001...", "Status__c": "Shipped" }
```

### Consumir en Apex Trigger

El trigger se ejecuta de forma **asíncrona** — en un contexto separado de quien publicó el evento.

```apex
trigger OrderShippedTrigger on OrderShipped__e (after insert) {
    for (OrderShipped__e event : Trigger.new) {
        System.debug('Orden enviada: ' + event.OrderId__c);
    }
}
```

**Puntos clave del trigger de Platform Events:**
- Solo `after insert` — no hay before, no hay update/delete.
- Contexto asíncrono — governor limits propios.
- Si el trigger falla, el evento se puede reintentar (ver `ReplayId`).

### Consumir en LWC con empApi

```js
import { LightningElement, wire } from 'lwc';
import { subscribe, unsubscribe, MessageContext } from 'lightning/empApi';

export default class OrderTracker extends LightningElement {
    subscription = {};

    connectedCallback() {
        subscribe('/event/OrderShipped__e', -1, (event) => {
            const payload = event.data.payload;
            this.orderId = payload.OrderId__c;
            this.status  = payload.Status__c;
        }).then(sub => { this.subscription = sub; });
    }

    disconnectedCallback() {
        unsubscribe(this.subscription, () => {});
    }
}
```

El segundo parámetro de `subscribe` es el `replayId`:
- `-1` → solo eventos nuevos desde ahora.
- `-2` → todos los eventos retenidos (hasta 72 horas).
- `{id}` → desde un evento específico.

### StandardVolume vs HighVolume

| Tipo | Publicaciones por hora | Retención | Transaccional |
|---|---|---|---|
| Standard Volume | 250,000 | 72 horas | Sí — se publica con el commit de la transacción |
| High Volume | Sin límite estricto (depende del org) | 72 horas | No — se publica independiente del commit |

> **Standard Volume:** si la transacción de Apex hace rollback, el evento NO se publica.  
> **High Volume:** el evento se publica siempre, aunque la transacción haga rollback.

## Change Data Capture (CDC)

Genera eventos automáticamente cada vez que un registro de Salesforce es creado, actualizado, eliminado o restaurado de la papelera. Sin código de publicación — Salesforce lo hace automáticamente.

### Habilitar CDC

Setup → Integrations → Change Data Capture → seleccionar los objetos.

O vía metadata:

```xml
<PlatformEventChannel>
    <channelType>data</channelType>
    <label>Account Change Events</label>
</PlatformEventChannel>
```

### Estructura del evento CDC

```json
{
  "schema": "...",
  "payload": {
    "ChangeEventHeader": {
      "entityName": "Account",
      "recordIds": ["001..."],
      "changeType": "UPDATE",
      "changedFields": ["Name", "Industry"],
      "changeOrigin": "com/salesforce/api/rest/59.0",
      "transactionKey": "...",
      "sequenceNumber": 1
    },
    "Name": "Acme Corp Updated",
    "Industry": "Technology"
  }
}
```

### changeType values

| Valor | Qué pasó |
|---|---|
| `CREATE` | Registro creado |
| `UPDATE` | Registro actualizado (solo campos modificados en payload) |
| `DELETE` | Registro eliminado |
| `UNDELETE` | Registro restaurado de papelera |
| `GAP_CREATE/UPDATE/DELETE` | Eventos perdidos — gap en la retención de 72h |

### Canales CDC

- Objeto estándar: `/data/AccountChangeEvent`
- Objeto custom: `/data/MyObject__ChangeEvent`
- Todos los objetos habilitados: `/data/ChangeEvents` (all-channel)

### Consumir CDC en Apex Trigger

```apex
trigger AccountCDCTrigger on AccountChangeEvent (after insert) {
    for (AccountChangeEvent event : Trigger.new) {
        EventBus.ChangeEventHeader header = event.ChangeEventHeader;

        if (header.changeType == 'UPDATE') {
            List<String> changedFields = header.changedFields;
            if (changedFields.contains('Industry')) {
                // Reaccionar al cambio de Industry
            }
        }
    }
}
```

### CDC vs PushTopic vs Platform Events

| Aspecto | PushTopic | CDC | Platform Events |
|---|---|---|---|
| Qué captura | Registros que coinciden con SOQL | Cambios en objetos habilitados | Eventos custom |
| Configuración | Código Apex | Setup (o metadata) | Metadata |
| Payload | Campos del SELECT | Solo campos modificados | Campos del evento |
| Retención | 24 horas | 72 horas | 72 horas |
| Estado | Legacy | Recomendado para cambios de datos | Recomendado para eventos de negocio |

## ReplayId y durabilidad

Cada evento en el bus tiene un `ReplayId` — un identificador secuencial. Permite al consumidor retomar desde el último evento procesado si estuvo desconectado.

```js
// Retomar desde el último evento procesado
const lastReplayId = getFromStorage(); // guardar en localStorage o Custom Setting
subscribe('/event/OrderShipped__e', lastReplayId, handleEvent);
```

El bus retiene eventos por **72 horas**. Si el consumidor estuvo más de 72h desconectado, puede recibir eventos GAP que le informan que hubo cambios que no puede recuperar.

## Backpressure y Ordering en eventos

### El problema de backpressure

Ocurre cuando el **consumidor procesa más lento de lo que el productor publica**. El bus acumula eventos sin procesar y, si el consumidor no alcanza, los eventos expiran a las 72 horas y se pierden.

**Síntomas en Salesforce:**
- El trigger de Platform Event acumula cola y llega con retraso creciente.
- El suscriptor externo no puede seguir el ritmo y empieza a recibir eventos GAP.

**Estrategias para manejar backpressure:**

| Estrategia | Descripción |
|---|---|
| **Throttling en el productor** | Publicar en lotes controlados en lugar de un evento por operación |
| **Batch processing en el consumidor** | El trigger de Platform Events puede procesar hasta 2,000 eventos por ejecución — aprovechar `Trigger.new` completo |
| **High Volume Events** | Para volúmenes muy altos donde Standard Volume se satura |
| **Separar consumidores lentos** | Si un consumidor es lento (ej. callout externo), sacarlo a un Queueable o Batch para no bloquear el trigger |

```apex
// Aprovechar el batch completo en el trigger — no procesar uno por uno
trigger OrderEventTrigger on OrderShipped__e (after insert) {
    List<Order__c> toUpdate = new List<Order__c>();
    for (OrderShipped__e ev : Trigger.new) {   // puede traer múltiples eventos
        toUpdate.add(new Order__c(Id = ev.OrderId__c, Status__c = ev.Status__c));
    }
    update toUpdate; // 1 DML para todo el lote
}
```

### El problema de ordering (orden de eventos)

**Salesforce NO garantiza el orden de entrega de Platform Events.** El bus entrega los eventos de forma aproximadamente ordenada pero puede variar bajo carga.

**Consecuencias:**
- Dos eventos para el mismo registro (`Status: Pending` → `Status: Shipped`) pueden llegar al revés.
- Si el consumidor aplica actualizaciones en el orden equivocado, el estado final es incorrecto.

**Estrategias para manejar ordering:**

| Estrategia | Descripción |
|---|---|
| **Timestamp en el payload** | Incluir `EventTime__c` en el evento y en el consumidor ignorar eventos más antiguos que el estado actual |
| **Número de secuencia** | Incluir un `SequenceNumber__c` y descartar eventos fuera de orden |
| **Last-write-wins con timestamp** | Solo aplicar el cambio si el timestamp del evento es mayor al `LastModifiedDate` del registro |
| **Idempotencia total** | Diseñar el handler para que aplicar el mismo evento N veces sea equivalente a aplicarlo una vez |

```apex
// Patrón: ignorar evento si el registro ya tiene un estado más reciente
trigger OrderEventTrigger on OrderShipped__e (after insert) {
    Set<Id> orderIds = new Set<Id>();
    Map<Id, OrderShipped__e> latestEvent = new Map<Id, OrderShipped__e>();

    for (OrderShipped__e ev : Trigger.new) {
        orderIds.add(ev.OrderId__c);
        latestEvent.put(ev.OrderId__c, ev); // si hay duplicados, el último gana
    }

    Map<Id, Order__c> existingOrders = new Map<Id, Order__c>(
        [SELECT Id, Status__c, LastModifiedDate FROM Order__c WHERE Id IN :orderIds]
    );

    List<Order__c> toUpdate = new List<Order__c>();
    for (Id orderId : latestEvent.keySet()) {
        OrderShipped__e ev = latestEvent.get(orderId);
        Order__c ord = existingOrders.get(orderId);
        // Solo actualizar si el evento es más reciente que el registro
        if (ord != null && ev.EventTime__c >= ord.LastModifiedDate) {
            toUpdate.add(new Order__c(Id = orderId, Status__c = ev.Status__c));
        }
    }
    if (!toUpdate.isEmpty()) update toUpdate;
}
```

### GAP events — pérdida de eventos

Cuando el consumidor estuvo desconectado más de 72 horas o el bus no pudo entregar todos los eventos, se reciben eventos tipo `GAP_CREATE`, `GAP_UPDATE`, `GAP_DELETE`.

**Estrategia:** ante un GAP, hacer una consulta directa a Salesforce para obtener el estado actual del registro, en lugar de intentar reconstruir el histórico de eventos perdidos.

## Resumen rápido

| Concepto | Clave |
|---|---|
| Platform Events | Mensajes custom; publicar con `EventBus.publish()`; trigger: solo `after insert` |
| Standard Volume | Transaccional — rollback cancela la publicación |
| High Volume | No transaccional — se publica siempre |
| CDC | Automático al modificar registros; payload solo incluye campos cambiados |
| `changeType` | `CREATE / UPDATE / DELETE / UNDELETE / GAP_*` |
| `ReplayId = -1` | Solo eventos nuevos. `-2` = todos los retenidos |
| `empApi` en LWC | Suscribirse a Platform Events y CDC desde la UI |
| Retención | Platform Events y CDC: 72 horas |
| Backpressure | Consumidor más lento que el productor — batch processing y throttling |
| Ordering | No garantizado — usar timestamp o secuencia para descartar eventos obsoletos |
| GAP events | Eventos perdidos — hacer consulta directa al estado actual del registro |
