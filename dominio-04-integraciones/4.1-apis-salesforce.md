# 4.1 — APIs de Salesforce (REST, SOAP, Bulk, Streaming)

---

## REST API

La API más usada. Basada en HTTP, formato JSON (y XML). Ideal para integraciones modernas y operaciones CRUD sobre registros.

### Endpoints base

```
https://[instancia].salesforce.com/services/data/v[version]/
```

### Operaciones CRUD

| Operación | Método HTTP | Endpoint |
|---|---|---|
| Consultar (SOQL) | GET | `/query?q=SELECT+Id+FROM+Account` |
| Obtener registro | GET | `/sobjects/Account/{Id}` |
| Crear | POST | `/sobjects/Account` |
| Actualizar (parcial) | PATCH | `/sobjects/Account/{Id}` |
| Actualizar completo | PUT | `/sobjects/Account/{Id}` |
| Eliminar | DELETE | `/sobjects/Account/{Id}` |
| Upsert por External ID | PATCH | `/sobjects/Account/ExternalId__c/{value}` |

### Autenticación

Siempre requiere un **Access Token** en el header:
```
Authorization: Bearer {access_token}
```

### Límites relevantes

- Hasta **2,000 registros** por respuesta de `/query` (paginación con `nextRecordsUrl`).
- **API requests diarios** según la edición y licencias (típico: 1,000 × nº de licencias).

## SOAP API

Basada en XML y WSDL. Más verbosa que REST pero más madura para integraciones enterprise con sistemas legacy (SAP, etc.).

### WSDLs disponibles

| WSDL | Uso |
|---|---|
| **Enterprise WSDL** | Específico del org — incluye todos los objetos y campos. Cambia si modificas el schema. |
| **Partner WSDL** | Genérico — funciona con cualquier org. Requiere trabajo extra para navegar el schema. |
| **Apex WSDL** | Para exponer métodos Apex `@WebService` como servicios SOAP. |
| **Metadata WSDL** | Para deployar/leer metadata programáticamente. |

### Operaciones clave

`login`, `query`, `queryMore`, `create`, `update`, `upsert`, `delete`, `merge`, `retrieve`, `describe`.

### Cuándo elegir SOAP sobre REST

- Integración con sistemas legacy que solo hablan SOAP.
- Necesitas operaciones como `merge` que REST no tiene.
- El sistema externo ya tiene un cliente SOAP generado desde el WSDL.

## Bulk API (v1 y v2)

Diseñada para procesar **millones de registros** de forma asíncrona. No usa límites de API estándar — tiene sus propios límites de Bulk.

### Bulk API 1.0 vs 2.0

| Aspecto | Bulk API 1.0 | Bulk API 2.0 |
|---|---|---|
| Formato | CSV, XML, JSON | Solo CSV |
| Jobs | Job → Batch → Result | Job → Upload → Result |
| Complejidad | Mayor (más pasos) | Menor (simplificado) |
| Ingest jobs | Sí | Sí |
| Query jobs | Sí | Sí |
| Uso recomendado | Cuando necesitas control granular de batches | Caso general, más simple |

### Flujo Bulk API 2.0 (Ingest)

```
1. POST /jobs/ingest          → Crear job (operación: insert/update/upsert/delete)
2. PUT /jobs/ingest/{jobId}/batches  → Subir CSV con los registros
3. PATCH /jobs/ingest/{jobId}  → Cerrar el job (state: UploadComplete)
4. GET /jobs/ingest/{jobId}    → Verificar estado (InProgress → JobComplete)
5. GET /jobs/ingest/{jobId}/successfulResults  → Descargar resultados exitosos
6. GET /jobs/ingest/{jobId}/failedResults      → Descargar registros fallidos
```

### Bulk Query

```
POST /jobs/query    { "query": "SELECT Id, Name FROM Account", "operation": "query" }
GET /jobs/query/{jobId}  → esperar JobComplete
GET /jobs/query/{jobId}/results  → descargar CSV
```

### Límites Bulk API

- Hasta **150 millones de registros** procesados por día (rolling 24h).
- Máx **10,000 jobs** activos simultáneamente.
- Máx **100 MB** por batch en Bulk 1.0.

## Streaming API

Permite que los clientes reciban notificaciones en tiempo real cuando ocurren cambios en Salesforce, sin hacer polling.

Basada en el protocolo **Bayeux/CometD** (long polling o SSE).

### Tipos de canales

| Canal | Qué notifica |
|---|---|
| **PushTopic** | Registros que coinciden con un SOQL query (legacy) |
| **Platform Events** | Eventos custom publicados por Apex, Flow, o API externa |
| **Change Data Capture (CDC)** | Cambios (create/update/delete/undelete) en objetos estándar y custom |
| **Generic Streaming** | Canal genérico para notificaciones arbitrarias |

### PushTopic (legacy, pero aún en exámenes)

```apex
PushTopic pushTopic = new PushTopic();
pushTopic.Name = 'AccountUpdates';
pushTopic.Query = 'SELECT Id, Name, Industry FROM Account';
pushTopic.ApiVersion = 59.0;
pushTopic.NotifyForOperationCreate = true;
pushTopic.NotifyForOperationUpdate = true;
pushTopic.NotifyForFields = 'Referenced'; // Solo campos en el SELECT
insert pushTopic;
```

El cliente se suscribe a `/topic/AccountUpdates`.

### Opciones de NotifyForFields

| Valor | Notifica cuando |
|---|---|
| `Referenced` | Cambia algún campo que está en el SELECT |
| `All` | Cambia cualquier campo del registro |
| `Select` | Solo si el registro entra/sale del filtro WHERE |
| `Where` | Solo si cambia un campo en el WHERE |

### Platform Events y CDC se cubren en detalle en el tema 4.4.

## Comparativa general de APIs

| API | Formato | Volumen | Sync/Async | Mejor para |
|---|---|---|---|---|
| REST | JSON/XML | Bajo-Medio | Síncrono | Integraciones modernas, CRUD |
| SOAP | XML | Bajo-Medio | Síncrono | Legacy, operaciones enterprise |
| Bulk | CSV | Masivo (millones) | Asíncrono | ETL, migración de datos |
| Streaming | JSON (CometD) | Eventos | Async push | Notificaciones en tiempo real |
| Metadata API | XML | Metadata | Async | Deploy, configuración |
| Tooling API | JSON | Metadata granular | Síncrono | IDE tools, debug |

---

## Resumen rápido

| Concepto | Clave |
|---|---|
| REST upsert | `PATCH /sobjects/Object/ExternalField__c/{value}` |
| SOAP Enterprise vs Partner | Enterprise: específico del org. Partner: genérico |
| Bulk API 2.0 flujo | Crear job → subir CSV → cerrar → esperar → descargar resultados |
| Streaming | CometD; canales: PushTopic, Platform Events, CDC, Generic |
| `NotifyForFields: Referenced` | Solo notifica si cambia un campo en el SELECT del PushTopic |
