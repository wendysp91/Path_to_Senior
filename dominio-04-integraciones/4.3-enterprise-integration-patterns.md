# 4.3 — Enterprise Integration Patterns (EIP)

---

## ¿Qué son los EIP?

Catálogo de patrones documentados por Gregor Hohpe y Bobby Woolf para diseñar integraciones entre sistemas. Salesforce los adopta en su arquitectura y los certifica en Integration Architect y CTA.

---

## Patrones de integración por estilo

### 1. Remote Process Invocation — Request/Reply (Síncrono)

Salesforce llama a un sistema externo y espera la respuesta en la misma transacción.

**Cuándo:** Necesitas el resultado inmediato para mostrar al usuario o continuar el proceso.

**Implementación en Salesforce:**
- Apex callout síncrono desde un botón o acción
- Named Credential + `Http.send()`

```
Usuario → [Salesforce] → HTTP POST → [Sistema Externo]
                      ←  Respuesta ←
```

**Limitación:** Sujeto al timeout de 120 segundos y governor limits síncronos.

---

### 2. Remote Process Invocation — Fire and Forget (Asíncrono)

Salesforce envía un mensaje al sistema externo sin esperar respuesta.

**Cuándo:** No necesitas la respuesta de inmediato. El procesamiento externo puede tardar.

**Implementación:**
- `@future(callout=true)` — simple, sin encolar
- `Queueable` con callout — cuando necesitas encadenar o monitorear
- Platform Events → sistema externo consume el evento

```
[Salesforce] → Evento / Mensaje → [Cola] → [Sistema Externo]
                                           (procesa cuando puede)
```

---

### 3. Batch Data Synchronization

Sincronización masiva de datos entre sistemas en lotes, típicamente en horarios definidos.

**Cuándo:** Grandes volúmenes que no requieren tiempo real. ETL nocturno, sincronización diaria.

**Implementación:**
- Bulk API desde el sistema externo
- Scheduled Apex + Batch Apex en Salesforce
- Heroku Connect para sincronización bidireccional con PostgreSQL

```
[Sistema Externo] →(nightly ETL)→ Bulk API → [Salesforce]
```

---

### 4. Remote Call-In (Sistema externo llama a Salesforce)

El sistema externo inicia la comunicación hacia Salesforce.

**Cuándo:** El sistema externo tiene datos que Salesforce necesita consumir, o necesita leer/escribir en Salesforce.

**Implementación:**
- REST API de Salesforce
- SOAP API
- Bulk API
- Apex REST (`@RestResource`) — endpoint custom en Salesforce
- Apex SOAP (`@WebService`) — servicio SOAP custom

```
[Sistema Externo] → REST/SOAP → [Salesforce REST/SOAP API o Apex REST]
```

---

### 5. UI Update Based on Data Change

La UI de Salesforce se actualiza en tiempo real cuando cambia un dato, sin que el usuario haga refresh.

**Cuándo:** Dashboards en tiempo real, notificaciones, progreso de procesos.

**Implementación:**
- Streaming API (Platform Events o CDC) + LWC que se suscribe
- `empApi` en LWC para suscribirse a Platform Events/CDC

```apex
// LWC suscribiéndose a Platform Events
import { subscribe } from 'lightning/empApi';
const channel = '/event/OrderShipped__e';
subscribe(channel, -1, (msg) => { this.status = msg.data.payload.Status__c; });
```

---

### 6. Data Virtualization (External Objects)

Acceder a datos externos desde Salesforce como si fueran objetos nativos, sin copiarlos.

**Cuándo:** Los datos viven en un sistema externo y no conviene replicarlos. Datos regulados, muy grandes, o que cambian frecuentemente.

**Implementación:**
- Salesforce Connect + External Data Source
- OData 2.0 / 4.0 adapter
- Custom Apex Adapter

## Patrones de Mensajería (Message Patterns)

### Point-to-Point

Un productor, un consumidor. Directo.

```
[Salesforce] → mensaje → [Sistema B]
```

### Publish-Subscribe

Un productor, múltiples consumidores. El productor no sabe quién consume.

```
[Salesforce] → Platform Event → [Sistema A]
                             → [Sistema B]
                             → [LWC en UI]
```

### Request-Reply

El productor envía y espera respuesta en un canal de retorno.

### Dead Letter Channel

Canal donde van los mensajes que no se pudieron procesar. En Salesforce: registros de error en Platform Events o manejo de `AllOrNoneHeader: false` en Bulk.

---

## Patrones de transformación

### Message Translator

Convierte el formato de un sistema al de otro. En Salesforce: Apex que transforma JSON externo al formato del sObject.

```apex
// Traducir JSON externo → Account
Map<String, Object> external = (Map<String, Object>) JSON.deserializeUntyped(body);
Account acc = new Account(
    Name        = (String) external.get('company_name'),
    Phone       = (String) external.get('phone_number'),
    ExternalId__c = (String) external.get('id')
);
```

### Content Enricher

Enriquece un mensaje con datos adicionales antes de procesarlo. Ej: el sistema externo manda solo el email del cliente → Salesforce busca el Contact y agrega su AccountId antes de continuar.

### Content Filter

Elimina campos del mensaje que no son necesarios para el consumidor.

### Splitter / Aggregator

- **Splitter:** Divide un mensaje grande en mensajes más pequeños (ej: un pedido con 100 líneas → 100 mensajes individuales).
- **Aggregator:** Junta múltiples mensajes en uno (ej: esperar todas las líneas antes de procesar el pedido completo).

---

## Patrones de routing

### Content-Based Router

Enruta el mensaje a diferentes destinos según su contenido.

```apex
if (payload.type == 'Order') {
    processOrder(payload);
} else if (payload.type == 'Invoice') {
    processInvoice(payload);
}
```

### Message Filter

Descarta mensajes que no cumplen ciertos criterios.

---

## Idempotencia

Un patrón crítico en integraciones: procesar el mismo mensaje más de una vez produce el mismo resultado que procesarlo una sola vez.

**Por qué importa:** Los sistemas de mensajería garantizan *at-least-once delivery* — el mismo mensaje puede llegar duplicado.

**Implementación:**
- Usar `upsert` con External ID en lugar de `insert`.
- Guardar un identificador único del mensaje procesado y verificar antes de procesar.

```apex
// Idempotente: upsert por External ID
Account acc = new Account(ExternalId__c = event.ExternalId, Name = event.Name);
upsert acc ExternalId__c;
```

## Resumen rápido

| Patrón | Clave |
|---|---|
| Request/Reply | Síncrono — espera respuesta en la misma transacción |
| Fire and Forget | Asíncrono — no espera respuesta; `@future`, Queueable, Platform Events |
| Batch Sync | ETL en lotes — Bulk API, Scheduled/Batch Apex |
| Remote Call-In | Sistema externo → Salesforce API o Apex REST |
| UI Update | Streaming API + `empApi` en LWC |
| Data Virtualization | External Objects — datos externos sin copiar |
| Publish-Subscribe | Platform Events — múltiples consumidores desacoplados |
| Idempotencia | `upsert` con External ID — procesar duplicados sin efectos secundarios |
