# 4.5 — Patrones de Datos en Integración

---

## External ID

Campo en Salesforce que almacena el identificador del registro en el sistema externo. Es la pieza central de cualquier integración bidireccional.

### Configuración

- Tipo: texto, número, email, o auto-number.
- Marcado como **External ID** en el field definition.
- Opcionalmente marcado como **Unique**.
- Salesforce crea un índice automático sobre él.

### Usos

**Upsert idempotente:**
```apex
Account acc = new Account(
    ExternalId__c = 'EXT-001',
    Name          = 'Acme Corp',
    Industry      = 'Technology'
);
upsert acc ExternalId__c;
```

**Lookup por External ID en REST:**
```http
PATCH /services/data/v59.0/sobjects/Account/ExternalId__c/EXT-001
{ "Name": "Acme Corp Updated" }
```

**Relacionar registros por External ID en Bulk API:**
```csv
Name,Account.ExternalId__c
"John Doe","EXT-001"
```
Salesforce resuelve el lookup al Account sin necesitar el Salesforce Id.

## Estrategias de sincronización

### Pull (Polling)

Salesforce o el sistema externo consulta periódicamente al otro para detectar cambios.

```apex
// Scheduled Apex que hace polling cada hora
public class ExternalSystemPoller implements Schedulable {
    public void execute(SchedulableContext sc) {
        HttpRequest req = new HttpRequest();
        req.setEndpoint('callout:ExternalSystem/api/changes?since=' + lastRunTimestamp);
        req.setMethod('GET');
        HttpResponse res = new Http().send(req);
        // procesar cambios...
    }
}
```

**Ventaja:** Simple, sin dependencia de que el externo soporte webhooks.  
**Desventaja:** Latencia alta, consume API calls aunque no haya cambios.

### Push (Event-Driven)

El sistema que origina el cambio notifica al receptor inmediatamente.

- Salesforce → Externo: Platform Event consumido por el externo, callout desde Apex.
- Externo → Salesforce: Webhook del externo → Apex REST en Salesforce.

**Ventaja:** Tiempo real, eficiente.  
**Desventaja:** Requiere que ambos sistemas soporten el mecanismo push.

### Bidireccional y gestión de conflictos

Cuando ambos sistemas pueden modificar el mismo dato:

| Estrategia | Descripción |
|---|---|
| Last Write Wins | El último en escribir gana. Simple pero puede perder cambios. |
| Source of Truth | Un sistema es el maestro; el otro solo lee. |
| Merge | Se combinan los cambios con lógica de negocio. Complejo. |
| Timestamp-based | Se comparan timestamps y gana el más reciente. |

**Patrón recomendado:** definir explícitamente qué sistema es el **master of record** para cada campo o entidad. Evitar que ambos sistemas modifiquen el mismo campo.

## Composite API

Permite combinar múltiples operaciones REST en una sola llamada HTTP, reduciendo el número de round-trips y aprovechando transaccionalidad.

### Composite Request (hasta 25 sub-requests)

```http
POST /services/data/v59.0/composite
{
  "allOrNone": true,
  "compositeRequest": [
    {
      "method": "POST",
      "url": "/services/data/v59.0/sobjects/Account",
      "referenceId": "newAccount",
      "body": { "Name": "Acme Corp" }
    },
    {
      "method": "POST",
      "url": "/services/data/v59.0/sobjects/Contact",
      "referenceId": "newContact",
      "body": {
        "LastName": "Doe",
        "AccountId": "@{newAccount.id}"   ← referencia al resultado anterior
      }
    }
  ]
}
```

Con `allOrNone: true`, si cualquier sub-request falla, toda la operación hace rollback.

### Batch Request (hasta 25 requests independientes)

Para operaciones que no dependen unas de otras y no requieren transaccionalidad:

```http
POST /services/data/v59.0/composite/batch
{
  "batchRequests": [
    { "method": "GET", "url": "/services/data/v59.0/sobjects/Account/001..." },
    { "method": "GET", "url": "/services/data/v59.0/sobjects/Contact/003..." }
  ]
}
```

### SObject Collections (hasta 200 registros, transaccional)

```http
POST /services/data/v59.0/composite/sobjects
{
  "allOrNone": false,
  "records": [
    { "attributes": { "type": "Account" }, "Name": "Acme", "ExternalId__c": "EXT-001" },
    { "attributes": { "type": "Account" }, "Name": "Beta", "ExternalId__c": "EXT-002" }
  ]
}
```

## Manejo de errores en integraciones

### Partial Success (allOrNone: false)

Permite que algunos registros fallen sin revertir los exitosos. Debes manejar los errores del response:

```apex
List<Database.UpsertResult> results = Database.upsert(records, ExternalId__c, false);
List<Account> failed = new List<Account>();
for (Integer i = 0; i < results.size(); i++) {
    if (!results[i].isSuccess()) {
        failed.add(records[i]);
        // loguear: results[i].getErrors()[0].getMessage()
    }
}
// retry o notificar
```

### Dead Letter Queue (DLQ) pattern

Registros que no se pudieron procesar van a una "cola de fallidos" para reintento manual o automático.

```apex
// Guardar en un objeto de error custom para reprocesar
Integration_Error__c err = new Integration_Error__c(
    Payload__c    = JSON.serialize(record),
    Error__c      = errorMessage,
    Retry_Count__c = 0,
    Status__c     = 'Pending'
);
insert err;
```

### Retry con backoff exponencial

```apex
Integer retryCount = error.Retry_Count__c.intValue();
if (retryCount < 3) {
    Integer delayMinutes = (Integer) Math.pow(2, retryCount); // 1, 2, 4 min
    // encolar con delay
    System.enqueueJob(new RetryJob(error.Id), delayMinutes);
}
```

## Data Mapping y Transformation

### Deserialización tipada vs no tipada

```apex
// No tipada — flexible pero requiere castings
Map<String, Object> data = (Map<String, Object>) JSON.deserializeUntyped(body);
String name = (String) data.get('company_name');

// Tipada — más segura si el schema es estable
ExternalOrder order = (ExternalOrder) JSON.deserialize(body, ExternalOrder.class);

public class ExternalOrder {
    public String order_id;
    public String status;
    public List<LineItem> items;

    public class LineItem {
        public String sku;
        public Integer quantity;
    }
}
```

### JSON con campos que son palabras reservadas de Apex

```apex
// 'type', 'class', 'object' son reservadas → usar deserialización no tipada o renombrar
Map<String, Object> parsed = (Map<String, Object>) JSON.deserializeUntyped(body);
String type = (String) parsed.get('type');
```

## Observabilidad end-to-end

Una integración sin observabilidad es una caja negra — no sabes si falla, cuándo falla ni por qué. La observabilidad cubre tres pilares: **logs, métricas y trazas**.

### Qué registrar en cada punto de la integración

```
[Sistema Externo] → [Salesforce Apex REST] → [Procesamiento] → [DML / callout]
       ↑                     ↑                      ↑                  ↑
  Log en externo       Log de entrada          Log de negocio     Log de resultado
```

**Mínimo recomendado por request:**
- Timestamp de inicio y fin.
- Identificador de correlación (Correlation ID) — mismo ID en todos los sistemas para poder trazar el flujo completo.
- Resultado: éxito / error + código de estado.
- Tiempo de respuesta.

### Correlation ID

El patrón más importante para observabilidad en integraciones. Un ID único que viaja con el mensaje a través de todos los sistemas, permitiendo reconstruir el flujo completo en los logs.

```apex
// Apex REST — leer el Correlation ID del header entrante o generar uno nuevo
@RestResource(urlMapping='/orders/*')
global with sharing class OrderService {

    @HttpPost
    global static String createOrder() {
        String correlationId = RestContext.request.headers.get('X-Correlation-Id');
        if (String.isBlank(correlationId)) {
            correlationId = generateCorrelationId();
        }

        // Propagar el Correlation ID al callout hacia el sistema siguiente
        HttpRequest req = new HttpRequest();
        req.setEndpoint('callout:InventorySystem/reserve');
        req.setHeader('X-Correlation-Id', correlationId);

        // Loguear con el Correlation ID
        logEvent('ORDER_RECEIVED', correlationId, 'success', null);

        return correlationId;
    }

    private static String generateCorrelationId() {
        return EncodingUtil.convertToHex(Crypto.generateAesKey(128)).substring(0, 16);
    }
}
```

### Dónde almacenar logs en Salesforce

| Opción | Pros | Contras |
|---|---|---|
| **Custom Object (Integration_Log__c)** | Visible en UI, reportable, sin límite especial | Consume almacenamiento, sujeto a governor limits del DML |
| **Platform Events (Log__e)** | No consume DML de la transacción principal, desacoplado | Solo 72h de retención en el bus; necesita consumidor |
| **Debug Logs** | Sin código adicional | Volátiles, no consultables vía SOQL, solo para dev |
| **Big Objects** | Sin límite práctico de almacenamiento, archivado | SOQL muy restringido, no se puede actualizar |

**Patrón recomendado: Platform Event para logs**

Publicar el log como Platform Event desde la transacción principal. Un trigger separado (contexto async) lo persiste en un Custom Object o Big Object. Esto evita que el DML del log afecte la transacción de negocio.

```apex
// Publicar log sin consumir DML de la transacción principal
Integration_Log__e logEvent = new Integration_Log__e(
    Correlation_Id__c = correlationId,
    Level__c          = 'ERROR',
    Message__c        = error.getMessage(),
    Endpoint__c       = endpoint,
    Duration_ms__c    = duration
);
EventBus.publish(logEvent); // no cuenta como DML en la transacción actual
```

### Métricas clave a monitorear

| Métrica | Cómo medirla en Salesforce |
|---|---|
| Tasa de éxito / error | Aggregate query sobre Integration_Log__c agrupado por status |
| Tiempo de respuesta promedio | `AVG(Duration_ms__c)` en Integration_Log__c |
| Registros en DLQ | Count de Integration_Error__c con Status = 'Pending' |
| Volumen de eventos publicados | EventBus.publish results |
| API usage diario | Setup → Company Information → API Requests Last 24 Hours |

### Alertas

```apex
// Alerta cuando la tasa de error supera el umbral
Integer errorCount = [
    SELECT COUNT() FROM Integration_Log__c
    WHERE Level__c = 'ERROR'
    AND CreatedDate = TODAY
];
if (errorCount > 100) {
    // Publicar Platform Event de alerta o enviar email al equipo
    EventBus.publish(new Alert__e(Message__c = 'Integration error rate exceeded threshold'));
}
```

### Trazabilidad distribuida

Para integrar con herramientas de observabilidad externas (Datadog, Splunk, New Relic):
- Enviar logs estructurados (JSON) vía callout a un endpoint de logging externo.
- Propagar siempre el `X-Correlation-Id` y el `X-Trace-Id` en los headers de todos los callouts.
- Usar el mismo Correlation ID en Salesforce, MuleSoft y el sistema destino.

## Resumen rápido

| Concepto | Clave |
|---|---|
| External ID | Base de la idempotencia; habilita upsert y lookup por valor externo |
| Pull / Push | Pull = polling; Push = evento/webhook en tiempo real |
| Master of Record | Definir qué sistema manda por cada campo — evitar conflictos |
| Composite API | Hasta 25 sub-requests encadenados con referencias (`@{ref.id}`) |
| `allOrNone: false` | Partial success — manejar explícitamente los errores |
| DLQ pattern | Guardar fallidos para reintento — no perder registros |
| Retry con backoff | Esperas exponenciales entre reintentos para no saturar el sistema externo |
| Correlation ID | ID único que viaja con el mensaje para trazar el flujo en todos los sistemas |
| Log via Platform Event | Publica el log sin consumir DML de la transacción principal |
| Métricas | Tasa de error, tiempo de respuesta, volumen en DLQ |
