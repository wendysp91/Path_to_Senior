# 4.7 — MuleSoft, Heroku Connect y External Objects

---

## MuleSoft

Plataforma de integración (iPaaS) de Salesforce. Actúa como middleware o ESB (Enterprise Service Bus) que conecta Salesforce con sistemas externos, transforma datos y orquesta flujos de integración.

### Cuándo usar MuleSoft en lugar de integración directa Apex

| Escenario | MuleSoft | Apex directo |
|---|---|---|
| Múltiples sistemas a conectar | ✅ MuleSoft como hub central | Complejo — demasiado código |
| Transformaciones complejas de datos | ✅ DataWeave (lenguaje de transformación) | Posible pero verboso |
| Integración con SAP, ERP legacy | ✅ Conectores pre-built | Complejo sin librerías |
| Orquestación multi-paso con reintentos | ✅ | Posible con Queueable/Batch |
| Integración simple Salesforce ↔ 1 sistema | Overkill | ✅ Apex callout |
| Control total en Salesforce | Parcial | ✅ |

### Conceptos clave de MuleSoft para exámenes de Salesforce

| Término | Descripción |
|---|---|
| **Anypoint Platform** | Plataforma completa de MuleSoft (diseño, deploy, monitoreo) |
| **Mule Application** | Aplicación de integración desplegada en MuleSoft |
| **Flow** | Secuencia de componentes que procesan un mensaje |
| **DataWeave** | Lenguaje de transformación de datos de MuleSoft |
| **Connector** | Plugin para conectarse a Salesforce, SAP, DB, etc. |
| **API-led Connectivity** | Arquitectura de 3 capas: System APIs → Process APIs → Experience APIs |

### API-led Connectivity

```
Experience APIs     → Lo que consume el frontend (app móvil, web)
        ↓
Process APIs        → Lógica de negocio y orquestación
        ↓
System APIs         → Conexión directa con cada sistema (Salesforce, SAP, DB)
```

Salesforce actúa como System API cuando MuleSoft se conecta a él como fuente de datos.

### Salesforce Connector en MuleSoft

MuleSoft incluye un Salesforce Connector que permite:
- SOQL queries.
- CRUD de registros.
- Publicar/consumir Platform Events.
- Suscribirse a CDC.
- Bulk API operations.

## Heroku Connect

Sincronización bidireccional entre Salesforce y una base de datos **PostgreSQL en Heroku**. No requiere código de integración — se configura desde una UI.

### Cómo funciona

```
[Salesforce] ←──── Heroku Connect ────→ [PostgreSQL en Heroku]
                  (sincronización
                   bidireccional)
```

- Crea tablas en PostgreSQL que mapean a objetos de Salesforce.
- Detecta cambios en Salesforce vía Streaming API/CDC y los replica en Postgres.
- Detecta cambios en Postgres y los replica en Salesforce vía REST API.

### Configuración

1. Crear una app en Heroku y agregar el add-on **Heroku Connect**.
2. Configurar la conexión OAuth con el org de Salesforce.
3. Mapear objetos de Salesforce a tablas de Postgres.
4. Definir dirección de sincronización: Salesforce→Postgres, Postgres→Salesforce, o bidireccional.

### Columnas especiales en Postgres

| Columna | Descripción |
|---|---|
| `sfid` | Salesforce Id del registro |
| `createddate` | Fecha de creación en Salesforce |
| `isdeleted` | Si el registro está en la papelera |
| `systemmodstamp` | Timestamp de última modificación |
| `_hc_lastop` | Última operación de Heroku Connect |
| `_hc_err` | Error de sincronización si aplica |

### Casos de uso

- Apps web en Heroku que necesitan leer/escribir datos de Salesforce.
- Reportes y análisis con herramientas que hablan SQL nativo (no SOQL).
- Procesamiento de datos en PostgreSQL con resultados sincronizados a Salesforce.

### Limitaciones

- No es tiempo real — hay latencia de segundos a minutos.
- No replica todos los objetos (depende de la licencia).
- Conflictos de escritura simultánea → Heroku Connect tiene reglas de resolución pero puede perder cambios.

## External Objects y Salesforce Connect

Permiten acceder a datos en sistemas externos como si fueran objetos nativos de Salesforce, sin copiarlos al org.

### ¿Qué es un External Object?

Un objeto cuyo almacenamiento está fuera de Salesforce. Se comporta como un sObject (aparece en Setup, puedes hacer SOQL restringido, crear relaciones), pero los datos viven en el sistema externo.

```
[Salesforce UI/Apex] → External Object → Salesforce Connect → [Sistema Externo]
                                          (OData Adapter)
```

### External Data Source

Configuración en Salesforce que define cómo conectarse al sistema externo.

| Adapter | Protocolo | Casos de uso |
|---|---|---|
| **OData 2.0** | OData REST | SAP, SharePoint, cualquier sistema OData |
| **OData 4.0** | OData REST | Versión más moderna |
| **Salesforce** | Salesforce API | Conectar dos orgs de Salesforce |
| **Custom Apex** | Tu código | Cualquier sistema con una API propia |
| **Simple URL** | HTTP | Datos simples vía URL |

### Custom Apex Adapter

Para sistemas que no hablan OData. Implementas la interfaz `DataSource.Provider` y `DataSource.Connection`:

```apex
global class MyExternalAdapter extends DataSource.Connection {
    override global List<DataSource.Table> sync() {
        // Definir la estructura (campos) del external object
        List<DataSource.Column> columns = new List<DataSource.Column>{
            DataSource.Column.text('ExternalId', 255),
            DataSource.Column.text('Name', 255),
            DataSource.Column.number('Amount', 16, 2)
        };
        return new List<DataSource.Table>{
            DataSource.Table.get('Orders', 'ExternalId', columns)
        };
    }

    override global DataSource.TableResult query(DataSource.QueryContext context) {
        // Hacer la llamada al sistema externo y devolver los datos
        HttpResponse res = makeCallout(context);
        List<Map<String, Object>> rows = parseResponse(res);
        return DataSource.TableResult.get(context, rows);
    }
}
```

### Relaciones con External Objects

| Tipo | Desde → Hacia | Descripción |
|---|---|---|
| **External Lookup** | Standard/Custom → External Object | El lookup field está en el objeto estándar |
| **Indirect Lookup** | External Object → Standard/Custom | Se basa en External ID (no en el Salesforce Id) |
| **Regular Lookup** | External Object → External Object | Entre dos external objects |

### Limitaciones de External Objects

- **No soportan**: SOQL con aggregate functions, `ORDER BY` en todos los casos, `UPDATE`/`DELETE` desde Apex (depende del adapter).
- **No se pueden usar en**: Workflow Rules, triggers sobre el external object directamente, reports complejos.
- **Performance**: Cada query al external object es un callout en tiempo real — latencia del sistema externo.
- **Governor limits**: Los callouts del External Object cuentan contra los límites de callout de la transacción.

## Cuándo usar cada tecnología

| Necesidad | Solución |
|---|---|
| Integración con 1 sistema externo, código en Salesforce | Apex callout + Named Credential |
| Múltiples sistemas, transformaciones complejas, reutilización | MuleSoft |
| App web en Heroku que necesita datos de Salesforce | Heroku Connect |
| Ver datos externos sin copiarlos, volumen bajo | External Objects (Salesforce Connect) |
| Migración masiva de datos | Bulk API |
| Eventos en tiempo real entre sistemas | Platform Events / CDC |

## Resumen rápido

| Concepto | Clave |
|---|---|
| MuleSoft | Middleware/iPaaS; API-led: System → Process → Experience APIs |
| DataWeave | Lenguaje de transformación de MuleSoft |
| Heroku Connect | Sync bidireccional Salesforce ↔ PostgreSQL; latencia segundos/minutos |
| `sfid` | Columna en Postgres con el Salesforce Id del registro |
| External Object | Datos externos visibles como sObjects sin copiarlos |
| OData Adapter | Protocolo estándar para External Objects; también Apex custom adapter |
| External Lookup | Lookup desde objeto estándar/custom → External Object |
| Indirect Lookup | Lookup desde External Object → objeto con External ID |
