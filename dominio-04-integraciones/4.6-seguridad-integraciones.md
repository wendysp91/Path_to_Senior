# 4.6 — Seguridad en Integraciones

---

## Connected Apps

Una Connected App define cómo una aplicación externa se autentica con Salesforce y qué puede hacer.

### Configuración clave

| Campo | Descripción |
|---|---|
| **Callback URL** | URL a la que Salesforce redirige después de OAuth Authorization Code |
| **OAuth Scopes** | Permisos que la app puede solicitar |
| **IP Relaxation** | Si se permite acceso desde cualquier IP o solo las permitidas del org |
| **Refresh Token Policy** | Cuánto tiempo es válido el refresh token |
| **Certificate** | Para JWT Bearer Flow |

### OAuth Scopes comunes

| Scope | Acceso |
|---|---|
| `api` | API REST, SOAP, Bulk, Streaming |
| `full` | Todo acceso (incluyendo `api`) |
| `refresh_token, offline_access` | Permite obtener refresh token |
| `openid` | OpenID Connect — incluye identidad del usuario |
| `chatter_api` | Solo Chatter API |
| `custom_permissions` | Permisos custom de la Connected App |
| `cdp_api` | Data Cloud |

### Gestionar acceso de Connected Apps

- **Policies:** Admin puede revocar acceso de una Connected App a usuarios específicos.
- **IP Ranges:** Restricción por IP en la Connected App o en el perfil del usuario.
- **Per-User Consent:** El usuario puede revocar el acceso desde sus Settings → Connected Apps.

## Integration User

Patrón de seguridad: crear un usuario dedicado para integraciones, no usar el admin.

**Por qué:**
- Auditoría clara — todos los cambios vía integración quedan en el nombre del integration user.
- Principio de mínimo privilegio — solo tiene acceso a lo que la integración necesita.
- Si las credenciales se comprometen, no expone una cuenta de admin.

**Configuración recomendada:**
- Perfil mínimo (Salesforce Integration User license si está disponible, o Minimum Access).
- Permission Sets con solo los objetos y campos que la integración necesita.
- No asignar a un humano — solo para uso programático.
- Activar **"Login IP Ranges"** en el perfil para restringir IPs si es posible.

## Seguridad de Apex REST

Cuando expones un servicio Apex REST (`@RestResource`), debes considerar:

### Autenticación

Todo request a la API de Salesforce requiere un Access Token válido. No hay endpoints de Apex REST sin autenticación (salvo Guest User en Experience Cloud, que es caso especial).

### Autorización — with sharing en Apex REST

```apex
@RestResource(urlMapping='/orders/*')
global with sharing class OrderService {  // ← with sharing aplica CRUD/FLS del usuario que llama

    @HttpGet
    global static List<Order__c> getOrders() {
        return [SELECT Id, Name FROM Order__c]; // respeta sharing rules
    }
}
```

### Validar el input — evitar injection

```apex
@HttpPost
global static String createOrder() {
    RestRequest req = RestContext.request;
    Map<String, Object> body = (Map<String, Object>) JSON.deserializeUntyped(req.requestBody.toString());

    // Validar campos obligatorios
    if (!body.containsKey('orderId') || body.get('orderId') == null) {
        RestContext.response.statusCode = 400;
        return 'orderId es requerido';
    }

    // No usar el input directamente en SOQL dinámico sin escapar
    String safeId = String.escapeSingleQuotes((String) body.get('orderId'));
}
```

### Evitar datos sensibles en respuesta

No devolver campos de seguridad, passwords ni PII innecesaria. Usar clases de respuesta custom en lugar de devolver el sObject completo:

```apex
global class OrderResponse {
    public String orderId;
    public String status;
    // No incluir campos internos ni de auditoría
}
```

## Seguridad de Callouts

### No hardcodear credenciales

```apex
// MAL ❌
req.setHeader('Authorization', 'Bearer eyJhbGciOiJSUzI1NiJ9...');

// BIEN ✅ — usar Named Credential
req.setEndpoint('callout:MiServicio/api/orders');
```

### Remote Site Settings / CSP Trusted Sites

Para hacer callouts a URLs externas, la URL debe estar en **Remote Site Settings** (Apex callouts) o **CSP Trusted Sites** (LWC, recursos estáticos).

```
Setup → Security → Remote Site Settings → New
URL: https://api.ejemplo.com
Active: ✅
```

Si el endpoint no está en Remote Site Settings, el callout falla con `Unauthorized endpoint`.

### Certificate Pinning (mTLS)

Para mayor seguridad, el servidor externo puede requerir que el cliente presente un certificado (mutual TLS). En Salesforce:

1. Subir el certificado del cliente a **Certificate and Key Management**.
2. Configurar la Named Credential para usar ese certificado como client certificate.

## Manejo de tokens — buenas prácticas

### Nunca almacenar tokens en campos de texto plano

- Usar **Named Credentials** — Salesforce cifra y gestiona el token.
- Si debes almacenarlos fuera, usar un campo encriptado (`Encrypted Text`).
- Nunca en código fuente, Custom Labels, o Custom Settings de texto plano.

### Expiración y revocación

- Access tokens expiran (típico: 1-2 horas).
- Named Credentials con OAuth gestionan el refresh automáticamente.
- Si sospechas compromiso: revocar desde la Connected App en Setup → rota las credenciales.

### Scopes mínimos necesarios

Solicitar solo los OAuth scopes que la integración realmente necesita. No usar `full` cuando `api` es suficiente.

## Sharing y visibilidad en integraciones

### with sharing vs without sharing en clases de integración

```apex
// Respeta las sharing rules del usuario que ejecuta el código
global with sharing class OrderService { }

// Ignora las sharing rules — ejecuta como sistema
global without sharing class OrderService { }
```

Para APIs internas donde el integration user ya tiene los permisos correctos configurados en su perfil/permission set, `with sharing` es la opción más segura.

### stripInaccessible

Filtra campos a los que el usuario no tiene acceso antes de devolver datos:

```apex
SObjectAccessDecision decision = Security.stripInaccessible(
    AccessType.READABLE,
    [SELECT Id, Name, SSN__c FROM Contact WHERE Id = :contactId]
);
List<Contact> safeContacts = decision.getRecords();
// SSN__c solo aparece si el usuario tiene acceso de lectura
```

## Resumen rápido

| Concepto | Clave |
|---|---|
| Connected App | Define cómo se autentica una app externa; gestiona scopes y policies |
| Integration User | Usuario dedicado, mínimo privilegio, auditoría clara |
| Apex REST + `with sharing` | Respeta sharing rules del usuario llamante |
| Remote Site Settings | Obligatorio para callouts a URLs externas |
| Named Credentials | No hardcodear tokens — Salesforce cifra y gestiona el refresh |
| `stripInaccessible` | Filtra campos no accesibles antes de devolver en respuesta |
| mTLS | Certificado de cliente para autenticación mutua en callouts |
| Scopes mínimos | Solicitar solo los permisos que la integración necesita |
